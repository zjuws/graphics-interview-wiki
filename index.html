请描述 GPU 渲染管线的完整流程
腾讯 字节 网易 必考
▼

📌 参考答案

GPU 渲染管线分为以下阶段：
  1. 应用阶段（CPU）：场景管理、剔除（视锥/遮挡）、Draw Call 提交
  2. 顶点着色器（Vertex Shader）：MVP 矩阵变换，将模型空间顶点转换到裁剪空间
  3. 曲面细分（Tessellation）（可选）：细分控制 + 细分求值，增加几何细节
  4. 几何着色器（Geometry Shader）（可选）：以图元为单位，可生成/删除图元
  5. 图元装配 & 裁剪：将顶点组装成三角形，对视锥外部分裁剪
  6. 光栅化（Rasterization）：三角形转换为片元（像素候选），插值重心坐标
  7. 片元着色器（Fragment Shader）：计算每个片元颜色，采样纹理、光照计算
  8. 逐片元测试：深度测试（Z-Test）、模板测试（Stencil Test）、混合（Blending）
  9. 帧缓冲输出：写入 Color Buffer，最终显示到屏幕
💡 面试加分点：提到 Early-Z（提前深度测试）可以减少 OverDraw，是性能优化重要手段。

MVP 矩阵分别代表什么？变换顺序是什么？
腾讯 通用 中频
▼

📌 参考答案

  • M（Model Matrix）：模型矩阵，将模型从本地空间变换到世界空间（包含平移/旋转/缩放）
  • V（View Matrix）：观察矩阵，将世界空间变换到相机空间（摄像机在原点，朝向 -Z）
  • P（Projection Matrix）：投影矩阵，将相机空间变换到裁剪空间（透视或正交投影）
变换顺序（右乘）：gl_Position = P × V × M × vertex
即：本地空间 → 世界空间 → 相机空间 → 裁剪空间 → NDC（除以w）→ 屏幕空间

什么是深度缓冲（Z-Buffer）？存在哪些精度问题？
字节 网易 中频
▼

📌 参考答案

Z-Buffer 是一块与屏幕同大小的缓冲区，存储每个像素当前最近片元的深度值（0~1）。在片元着色后与当前 Z-Buffer 中的值比较，更近则写入颜色并更新深度值。

精度问题：
  • Z-Fighting（深度冲突）：两个面距离很近时，浮点精度不足导致交替显示
  • 精度分布不均匀：透视投影后深度在近裁面附近精度高，远处精度急剧下降
解决方案：
  • 使用 Reversed-Z（反转深度缓冲），利用浮点数在0附近精度高的特性
  • 增大近裁面距离，缩小 Far/Near 比值
  • 使用 32-bit 深度缓冲代替 24-bit

什么是 Draw Call？如何减少 Draw Call 提升性能？
腾讯 字节 高频
▼

📌 参考答案

Draw Call 是 CPU 向 GPU 发出的一次绘制命令，每次调用都涉及状态切换和命令提交，CPU 开销较大。

减少 Draw Call 的主要手段：
  • 合批（Batching）：静态合批（Static Batching）/ 动态合批（Dynamic Batching）将多个使用相同材质的物体合并成一次绘制
  • GPU Instancing：相同网格/材质的物体通过实例化渲染，一次 Draw Call 绘制多个实例
  • 图集（Texture Atlas）：合并多张小纹理为一张，减少材质切换
  • Indirect Rendering：GPU 驱动渲染，CPU 不再逐个提交

什么是 Mesh Shader？它如何取代传统顶点/几何着色器流水线？
字节 腾讯 进阶
▼

📌 参考答案

Mesh Shader（网格着色器） 是 DirectX 12 Ultimate / Vulkan 1.3 引入的新型几何处理阶段，完全替代 Vertex Shader + Hull/Domain Shader + Geometry Shader 的传统流水线。

架构对比：
  • 传统管线：CPU 准备顶点数据 → VB/IB → VS → （可选 HS/DS/GS） → 光栅化，瓶颈在 CPU Draw Call 和固定功能装配阶段
  • Mesh Shader 管线：Task Shader（可选，决定派发多少 Mesh 工作组） → Mesh Shader（直接在 GPU 上生成顶点和图元） → 光栅化
核心优势：
  • GPU 驱动几何：Mesh Shader 在 GPU 端直接生成几何，可实现 GPU Culling、LOD、Procedural Geometry，消除 CPU 瓶颈
  • 更灵活的输出：可输出任意顶点+图元布局，不再受固定 IA 约束
  • 高效剔除：配合 Task Shader 可在 GPU 端做 Cluster 级 / Meshlet 级的 Frustum + Backface + Occlusion 剔除
Meshlet：通常每个 Meshlet 包含 64~128 个顶点和 64~128 个图元，是 Mesh Shader 的基本处理单元，类似于 Compute Shader 的线程组。Nanite（UE5）的核心几何处理正是基于此思想。
💡 面试要点：重点说明 Task Shader + Mesh Shader 的两阶段结构，以及与传统 IA（Input Assembler）流水线的核心差异。UE5 Nanite 是最典型的工程案例。

什么是 GPU 驱动渲染（GPU-Driven Rendering）？Indirect Draw 如何实现？
腾讯 字节 进阶
▼

📌 参考答案

GPU 驱动渲染（GPU-Driven Rendering） 指将剔除、排序、Draw Call 生成等传统 CPU 工作完全移交 GPU 执行，CPU 仅提交少量命令。

关键 API：
  • DrawIndirect / DrawIndexedIndirect：Draw Call 的参数（顶点数、实例数、偏移等）存储在 GPU Buffer 中，GPU 可在 Compute Shader 中动态写入，CPU 仅调用一次 Indirect 命令
  • ExecuteIndirect（DX12）/ vkCmdDrawIndirectCount（Vulkan）：支持变长数量的 Indirect Draw，可在 GPU 上确定实际绘制数量
典型流程：
  1. CPU 将所有物体的实例数据（变换矩阵、材质 ID 等）上传到 GPU Buffer
  2. Compute Shader 执行 GPU Culling（视锥 + Hi-Z 遮挡剔除），将通过剔除的实例写入 Indirect Draw Buffer
  3. 调用 DrawIndexedIndirect，GPU 根据 Buffer 批量绘制所有可见实例，通常一次 Draw Call 即可
优势：极大减少 CPU-GPU 通信，解放 CPU，适合大规模植被、建筑等场景。《地平线：零之曙光》《荒野大镖客2》均采用此方案。
💡 配套技术：Hi-Z（Hierarchical Z）用于 GPU 端的遮挡剔除，与 GPU-Driven Rendering 几乎绑定使用，面试时可主动提及。

Mesh Shader 与传统顶点/几何着色器管线有什么区别？它解决了哪些性能瓶颈？
字节 腾讯 高频
▼

📌 参考答案

传统管线的瓶颈：
  • 顶点着色器以单顶点为单位，IA（Input Assembler）阶段从显存获取顶点数据是串行瓶颈
  • 几何着色器（GS）性能差，GPU 对其并行度优化有限，大量生成图元时吞吐下降严重
  • 整个图元剔除（Primitive Culling）必须在顶点着色后进行，无效顶点已经消耗了计算资源
Mesh Shader（DX12 Ultimate / Vulkan 1.3+，NVIDIA Turing+）架构：
  1. Task Shader（Amplification Shader）：以工作组（Workgroup）为单位运行，决定生成多少个 Mesh Shader 线程组，可在此阶段做粗粒度剔除（如背面/视锥剔除），输出 payload 给 Mesh Shader
  2. Mesh Shader：每个线程组输出最多 256 个顶点和 512 个图元，可以直接输出任意拓扑结构（不受三角带/扇形限制），最终送入光栅化阶段
优势：
  • 消除 IA 瓶颈：Mesh Shader 直接从任意 Buffer 读取数据（类似 Compute Shader），绕过固定功能的 IA 阶段
  • 高效 Meshlet 剔除：配合 Meshlet（每个 Meshlet 约 64~128 个顶点、126 个三角形），可在 Task Shader 中并行剔除无效 Meshlet，大幅减少光栅化开销
  • 替代 GS：Mesh Shader 对 GPU 并行更友好，可完全替代几何着色器用于粒子、草地等大量图元生成场景
实际应用：UE5 Nanite 的 Software Rasterizer 充分利用了 Mesh Shader 的 Meshlet 架构；《战地 2042》等 AAA 游戏用于大规模植被渲染。
💡 记忆口诀：Mesh Shader = Compute Shader 的并行能力 + 光栅化的输出能力。Task Shader 负责"要不要画"，Mesh Shader 负责"怎么画"，一起替代了 IA + VS + GS 这段管线。

UE5 Nanite 虚拟几何体的核心架构是什么？它如何做到亿级三角形的实时渲染？
腾讯 字节 必考
▼

📌 参考答案

Nanite 核心架构：
  • Meshlet（簇）化：将网格离线拆分为约 128 个三角形一组的 Cluster，并构建层次化有向无环图（Cluster DAG），每个簇记录其父子关系和简化误差
  • GPU Driven LOD 选择：运行时在 GPU Compute Shader 内完成视锥体裁剪、HZB 遮挡剔除和 LOD 选择，CPU 几乎不参与——选择 DAG 上满足"屏幕误差 < 1 像素"的一个"剪切面"来绘制
  • Software Rasterizer：对亚像素级别的小三角形（边长 < 约 2 像素），用 Compute Shader 自实现软件光栅化，比硬件光栅器效率更高（硬件处理小三角形开销大）；大三角形仍走硬件光栅化
  • Visibility Buffer（可见性缓冲）：光栅化输出每像素的 VisBuffer（Instance ID + Cluster ID + 三角形 ID），延迟到 Material Pass 才根据 VisBuffer 索引材质并着色，避免 GBuffer 带宽爆炸
  • 流式加载：Meshlet 数据高度压缩并支持细粒度按需流入显存，不依赖 CPU 端 LOD 切换
性能关键点：
  • 整条管线完全 GPU Driven，CPU 只提交一次 Indirect Dispatch/Draw，消除了百万级 Draw Call 的 CPU 瓶颈
  • 每个 Cluster 独立剔除，剔除粒度从"物体"细化到"128 三角形组"，剔除率大幅提升
  • 局限：不支持骨骼动画蒙皮（Skinned Mesh）、不支持 World Position Offset 复杂变形；透明物体不走 Nanite
💡 面试建议：Nanite = Cluster DAG + GPU Driven 剔除 + Software Rasterizer + Visibility Buffer，这四点说清楚即可拿满分。特别强调"软件光栅化处理小三角形"是 Nanite 区别于传统方案的工程亮点。

GPU Driven Rendering（GPU 驱动渲染）的核心思想是什么？Indirect Draw Call 如何减少 CPU-GPU 通信开销？
腾讯 网易 中频
▼

📌 参考答案

传统渲染瓶颈：CPU 遍历场景所有物体 → 视锥剔除 → 逐物体提交 Draw Call → GPU 绘制。当场景物体超过数万个时，CPU 端剔除和 Draw Call 提交成为瓶颈（CPU 带宽约 100 万次/秒）。

GPU Driven 核心思想：将剔除、LOD 选择、Draw Argument 生成全部移到 GPU Compute Shader 内完成，CPU 只提交一次 DrawIndexedIndirect，实际绘制参数从 GPU Buffer 中读取。
  • ExecuteIndirect / MultiDrawIndirect：DX12/Vulkan 的 Indirect Draw API，允许 GPU Buffer 直接指定绘制参数（顶点起始/数量/实例数），CPU 不参与每帧的绘制决策
  • GPU Culling Pipeline：① Compute Pass 读取所有物体 AABB → 视锥剔除 → 遮挡剔除（HZB）→ 将通过剔除的物体写入 Compact Buffer；② Indirect Draw 从 Compact Buffer 中按需绘制
  • Two-Phase Occlusion Culling：第一帧用上一帧 HZB 剔除（可能漏画）→ 渲染可见集合 → 更新 HZB → 第二 Pass 检查被错误剔除的物体并补渲染（Reprojection Test）
收益量化：Draw Call 从 O(场景物体数) 降低到 O(材质批次数)，百万物体场景 GPU 利用率可从 40% 提升到 90% 以上。
💡 工程细节：GPU Driven 需要场景所有物体的数据（变换矩阵、AABB、LOD 信息）提前上传到 GPU Buffer，并按材质/管线状态分组（以减少 PSO 切换），这是引擎级别的架构改造，而非简单的 API 调用优化。

光线追踪的基本流程是什么？与光栅化的核心区别在哪里？
腾讯 字节 网易 必考
▼

📌 参考答案

光线追踪（Ray Tracing）基本流程：
  1. 光线生成（Ray Generation）：从相机出发，为每个像素生成一条主光线（Primary Ray）
  2. 光线求交（Intersection）：遍历场景几何体，找到光线与场景的最近交点
  3. 着色（Shading）：在交点处计算材质、发出阴影光线（Shadow Ray）向光源检测遮挡
  4. 递归追踪（Recursive Tracing）：如有反射/折射，生成次级光线继续追踪，直到达到递归深度或能量阈值

与光栅化的核心区别：
对比维度 光栅化 光线追踪
流程方向几何 → 像素像素 → 几何
全局光照近似（SSAO、IBL）天然支持
反射/折射需要 SSR/环境贴图近似递归追踪，物理准确
实时性能极高（GPU 专为此优化）较低（需 RT Core 加速）
硬件依赖所有 GPU需 DXR / Vulkan RT / RT Core

现代游戏多采用混合渲染（Hybrid Rendering）：光栅化主视图，光线追踪负责阴影、反射、AO 等高质量效果。
💡 面试要点：能说出"光栅化是从几何到像素，光追是从像素到几何，方向相反但可以融合"，显示出对两者本质差异的理解。

BVH 加速结构的原理是什么？SAH 策略如何优化构建质量？
腾讯 字节 网易 必考
▼

📌 参考答案

BVH（Bounding Volume Hierarchy，层次包围盒） 是光线追踪中最常用的加速结构，将场景几何体组织成树状结构，快速剔除大量不需要求交的图元。

构建过程：
  1. 为每个图元（三角形）计算 AABB（轴对齐包围盒）
  2. 递归地将图元集合分为两组，为每组建立子节点 AABB
  3. 重复直到叶节点图元数量低于阈值（通常 1~4 个）
划分策略：
  • 中点划分：沿最长轴从中点切分，简单快速但质量差
  • SAH（Surface Area Heuristic，表面积启发式）：
    • 代价函数：Cost = C_traversal + p_left × N_left × C_intersect + p_right × N_right × C_intersect
    • p 为子节点 AABB 面积与父节点的比率，遍历所有候选分割点选最小代价
    • 生成的树更平衡，遍历效率最优，是工业界标准（OptiX、DXR 均采用）
遍历：光线与根节点 AABB 测试，命中则递归测试两个子节点，未命中剪枝，到达叶节点时进行光线-三角形求交（Möller–Trumbore 算法）。

BLAS / TLAS（DXR/Vulkan RT）：
  • BLAS（Bottom-Level AS）：每个 Mesh 的几何 BVH，由 GPU 驱动构建
  • TLAS（Top-Level AS）：包含所有实例的 BVH，每帧根据实例变换更新（Refit），无需重建 BLAS
💡 面试加分：BLAS/TLAS 两级结构是实时光追工程中的必考点，能说清两级的分工（Mesh 几何 vs 场景实例）是进阶得分关键。

ReSTIR 算法的原理是什么？它如何解决实时全局光照的采样效率问题？
腾讯 字节 进阶
▼

📌 参考答案

ReSTIR（Reservoir-based Spatiotemporal Importance Resampling） 是 NVIDIA 2020 年提出的实时光照采样算法，通过时域和空域重用样本，显著减少每像素所需的光源采样数。

核心思路：
  • Reservoir（蓄水池采样）：每个像素维护一个蓄水池，记录当前最优候选光源及权重之和，支持 O(1) 更新
  • 时域重用（Temporal Reuse）：当前帧的候选样本与上一帧的 Reservoir 合并，有效样本数翻倍
  • 空域重用（Spatial Reuse）：借用邻域像素（通常 5 个随机邻居）的 Reservoir，进一步扩大有效采样数
效果：每像素仅 1-2 条光线，但等效采样数可达数百条，特别适合大量动态光源场景（RTXDI 基于此原理）。

注意事项：时域重用需要 Visibility Reuse（重新验证历史样本可见性），否则快速遮挡变化时会产生 Bias；实践中使用 Unbiased Reservoir 确保收敛结果无偏。
💡 RTXDI（NVIDIA RTX Direct Illumination）是 ReSTIR DI 的工程实现，已用于 《赛博朋克 2077》《战地》等 AAA 游戏，是学习实时光追必看案例。

SDF（Signed Distance Field）光线步进在软阴影和全局光照中如何应用？
腾讯 网易 进阶
▼

📌 参考答案

SDF（有向距离场） 存储空间中每个采样点到最近表面的有符号距离（内部为负，外部为正），可用于快速的光线步进（Ray Marching）。

Sphere Tracing 流程：
  1. 从光线起点出发，每步沿光线方向前进 SDF(P) 距离（安全步长，不会穿越表面）
  2. 若 SDF(P) < ε（接触阈值），则认为光线命中表面
  3. 若步进次数超出最大值或超出最大距离，则认为未命中
软阴影应用：在步进过程中记录 min(SDF(P) / t)（t 为当前步进距离），该最小值反映光线与遮挡体的最近接近程度，可直接用作软阴影的遮挡因子，不需要额外采样。

Lumen（UE5）中的应用：
  • 场景被预计算为全局 SDF（Mesh Distance Field + Global SDF），分辨率逐级降低
  • 光线追踪阶段用 Global SDF 进行远距离 Sphere Tracing，用精细 Mesh SDF 处理近距离求交，无需 BVH
  • 结合 Radiance Cache 和屏幕空间探针，实现实时全局光照

问题背景：场景中存在数百万个 Emissive 三角形（面光源）时，每像素只能追少量光线，如何高效选出"贡献大"的光源样本是核心挑战。朴素均匀采样在光源密集场景中方差极大，渲染结果噪点严重。

ReSTIR（Reservoir-based SpatioTemporal Importance Resampling，NVIDIA 2020）核心思想：
  1. Reservoir 采样（WRS）：每像素维护一个 Reservoir（蓄水池），通过加权蓄水池采样从候选光源集中选出 1 个样本，记录其权重和贡献，保证采样概率与重要性成正比
  2. 时域复用（Temporal Reuse）：将上一帧的 Reservoir 样本通过运动向量重投影到当前帧，与当前帧新样本合并更新 Reservoir，等效于积累了多帧的样本数（可积累数十帧 × N 个初始候选 = 数千个有效样本）
  3. 空域复用（Spatial Reuse）：从邻域像素（半径内随机选 K 个）读取其 Reservoir，合并到当前像素的 Reservoir，进一步降低方差
数学保证：通过 MIS（多重重要性采样）权重修正，RIS（Resampled Importance Sampling）保证合并后采样结果仍趋于无偏估计。

效果：在 100 万个面光源场景下，每像素只需追 1 条阴影光线，质量接近每像素数千次路径追踪采样的结果；实时帧率可行（配合 DLSS 降噪）。

变体：ReSTIR GI（间接光照复用）、ReSTIR PT（完整路径追踪样本复用）、NVIDIA RTX Remix 和 Cyberpunk 2077 RT 模式均使用 ReSTIR 系列技术。
💡 关键点：ReSTIR 的本质是"以时空邻域的历史样本换取高质量采样"，核心数据结构 Reservoir 只存储 1 个当前最优样本 + 其归一化权重，内存开销极低，这是它能用于实时渲染的工程关键。

  • 高质量模式：硬件光线追踪——在支持 DXR/Vulkan RT 的 GPU 上，使用硬件 BVH 进行精确光线求交，质量更高，适合 RTX 4090 等旗舰显卡

  • Surface Cache 读取间接光照时，先查询 Radiance Cache，减少多次反弹追踪的开销
  • 通过时间积累（Temporal Accumulation）平滑更新，避免每帧重建带来的噪声

  • 完全动态的全局光照延迟（Light Bounce 延迟约 1-2 帧），快速光照变化场景可能出现滞后感
  • 不支持 Nanite 网格的精确反射（需降级为 Screen Space Reflection 或 Reflection Capture）

什么是 Bindless Rendering？与传统 Descriptor Set 绑定相比有何优势？Vulkan 的 Descriptor Indexing 扩展如何实现 Bindless？
腾讯 网易 中频
▼

📌 参考答案

传统 Descriptor 绑定的痛点：
  • 在传统 Vulkan/DX12 工作流中，每个 Draw Call 往往需要重新绑定 Descriptor Set（切换 Texture、Buffer 等），产生大量 Descriptor Update / Set 切换 开销
  • Descriptor Set Layout 的 Slot 数量有上限，复杂场景（数千张纹理）难以用有限的绑定点覆盖
  • CPU 驱动侧需要不断 vkCmdBindDescriptorSets，阻碍多线程提交和 GPU Driven 渲染
Bindless Rendering 核心思想：
  • 将场景中所有纹理/Buffer 一次性注册到一个超大 Descriptor Array（如 65536 个 Texture Slot），Shader 通过动态索引（Index）而非静态绑定来访问资源
  • 每帧只需绑定这一个超大 Descriptor Set，Draw Call 通过 Push Constant 或 Storage Buffer 传入每个 Draw 使用的资源 Index，完全避免 Descriptor Set 切换
Vulkan Descriptor Indexing 实现（VK_EXT_descriptor_indexing，Vulkan 1.2 核心）：
  • RuntimeDescriptorArray：允许 GLSL/HLSL 中声明无上界数组 layout(set=0, binding=0) uniform sampler2D textures[];，数组大小在运行时确定
  • PartiallyBound：Descriptor Array 中可以有未初始化的 Slot（Shader 保证不访问这些 Slot 即可），方便按需填充
  • UpdateAfterBind：允许在 Command Buffer 录制后继续更新 Descriptor，支持流式加载新纹理而无需重录 CB
  • NonUniformResourceIndex：动态索引纹理时，硬件可能需要非统一地址访问，需用 nonuniformEXT() 装饰防止索引被错误 promote 为统一值
与 GPU Driven 渲染的配合：
  • GPU 通过 Indirect Draw 执行 Draw Call，每个 Draw 的参数（包括 Material Index / Texture Index）都存储在 GPU 端 Buffer，CPU 零介入
  • Bindless + Indirect Draw + GPU Culling 是现代引擎（UE5 Nanite、Unity SRP GPU Resident Drawer）高性能渲染管线的三大支柱
💡 面试要点：区分 Bindless 的两个层次：① 概念层——减少 Descriptor Set 切换；② 实现层——Descriptor Indexing 扩展的 4 个关键特性（RuntimeArray / PartiallyBound / UpdateAfterBind / NonUniformIndex）。能说清这两层就足以脱颖而出。

Vulkan RenderPass 与 Subpass 的设计意图是什么？为何对移动端 TBDR GPU 特别重要？Subpass 如何避免带宽浪费？
腾讯 网易 必考
▼

📌 参考答案

TBDR（Tile-Based Deferred Rendering）GPU 架构回顾：
  • 移动端 GPU（Mali/Adreno/Apple M 系列）采用 TBDR：将屏幕分为 N×N 像素的小 Tile，所有 Draw Call 的几何先在 Vertex 阶段处理完毕并存入 Parameter Buffer，再按 Tile 逐块执行光栅化和片段着色
  • 每个 Tile 有一块 On-Chip Tile Memory（片上缓存），容量小但速度极快（相当于 SRAM），Tile 内的 Color/Depth/Stencil 数据全程在片上处理，只在 Tile 完成后才 Resolve 写入 System Memory（DRAM）
  • 如果算法需要跨 Tile 读取上一个 Pass 的结果（如延迟着色的 GBuffer），数据必须从 DRAM 重新读取 → 巨大带宽浪费，极耗移动端电量
Vulkan RenderPass / Subpass 如何解决：
  • RenderPass：声明整个渲染过程所需的 Attachment（Color/Depth 等），以及 loadOp/storeOp（是否从 DRAM 加载或写回 DRAM）
  • Subpass：RenderPass 内部可划分多个 Subpass，每个 Subpass 声明输入（inputAttachment）和输出（colorAttachment），驱动可据此判断 GBuffer 数据是否需要出 Tile 或可以全程留在 On-Chip Memory 内
  • 延迟渲染典型用法：
    • Subpass 0（GBuffer Pass）：输出 Albedo/Normal/Depth 到片上，storeOp = DONT_CARE（不写回 DRAM）
    • Subpass 1（Lighting Pass）：inputAttachment 读取 Subpass 0 的 GBuffer（从 Tile Memory 读，不过 DRAM），输出最终颜色
  • storeOp = DONT_CARE：告知驱动该 Attachment 处理完 Tile 后可直接丢弃，节省 Tile→DRAM 写回带宽
  • loadOp = DONT_CARE：Tile 开始时不从 DRAM 加载，直接以未定义内容初始化（适合全屏 Clear 的场景）
Subpass Dependency：定义 Subpass 之间的执行和内存依赖（类似 Pipeline Barrier），确保 Subpass 0 写完 GBuffer 后 Subpass 1 才读取，且该依赖在 On-Chip 内完成而无需 Full Memory Barrier。

桌面端（IMR GPU）的取舍：桌面 GPU（NVIDIA/AMD）无 Tile Memory，Subpass 的带宽优化基本无效，驱动通常将多 Subpass 退化为普通 Pass；但语义上仍正确，代码可跨平台。
💡 记忆口诀：TBDR = 分块 + 片上缓存；Subpass = 让 GBuffer 留在片上不出 DRAM；storeOp/loadOp DONT_CARE = 告诉驱动省掉无用带宽。这三句话串起整个移动端渲染优化的核心思路。

BVH（层次包围体）加速结构的构建原理是什么？SAH（表面积启发式）如何指导最优分割？DXR/Vulkan RT 中 TLAS 与 BLAS 各自的职责是什么？
腾讯 网易 中频
▼

📌 参考答案

BVH 基本原理：
  • BVH（Bounding Volume Hierarchy）是一棵二叉树，每个节点保存一个轴对齐包围盒（AABB），叶节点包含实际三角形图元
  • 光线与场景求交时，从根节点出发，若光线与节点 AABB 不相交则剪枝整个子树；否则递归检测子节点，直至叶节点与三角形精确求交
  • 时间复杂度从 O(N) 降至 O(log N)（理想平衡树），典型场景 100 万三角形只需约 20 次 AABB 测试
BVH 构建方法：
  • 中心点分割（Median Split）：按某轴中点将图元分为两半，简单但质量差，AABB 重叠多
  • SAH（Surface Area Heuristic，表面积启发式）：
    • 目标：最小化期望的光线-三角形相交测试次数
    • 代价公式：Cost = C_trav + (SA_left/SA_parent) × N_left × C_isect + (SA_right/SA_parent) × N_right × C_isect
    • 其中 SA_left/SA_parent 是子节点与父节点表面积之比（代表光线命中概率），N 是图元数，C_trav/C_isect 是遍历/相交的相对开销
    • 枚举 K 个分割平面候选（Binning SAH），选代价最小的分割 → 质量最优的 BVH 构建
  • HLBVH（Morton 码并行构建）：将图元中心坐标映射到 Morton 码（Z 序曲线），并行排序后快速构建 BVH，适合 GPU 端实时重建（如粒子/软体动画场景）
DXR/Vulkan Ray Tracing 中的两级结构：
  • BLAS（Bottom-Level Acceleration Structure）：对单个 Mesh 的三角形构建的 BVH，包含模型的局部坐标系几何数据；静态 Mesh 的 BLAS 只需构建一次（VK_BUILD_ACCELERATION_STRUCTURE_PREFER_FAST_TRACE_BIT）
  • TLAS（Top-Level Acceleration Structure）：管理多个 BLAS 实例，每个实例包含 BLAS 引用 + 4×3 变换矩阵；动态场景只需每帧重建 TLAS（Rebuild），静态 BLAS 不变
  • TLAS 的优势：① 支持同一 BLAS 多实例（Instance）复用，无需复制三角形数据；② TLAS 重建代价远低于 BLAS，适合频繁变化的场景（动画角色更新 BLAS，场景图更新 TLAS）
  • Compaction：Build 后执行 BLAS Compaction，移除构建过程中的临时内存，通常可减少 30-50% VRAM
💡 面试要点：BVH + SAH = 光线追踪的"索引+代价优化"二元组；TLAS/BLAS 分层设计 = 让实例复用和增量更新成为可能。这两点是 DXR/Vulkan RT 硬件加速的核心设计哲学。

屏幕空间反射（SSR）的完整实现流程是什么？线性 Ray March 与 Hi-Z Tracing 有何本质区别？SSR 的主要局限性如何处理？
腾讯 字节 必考
▼

📌 参考答案

SSR 的基本思路：
  • 利用已渲染完成的 Color Buffer、Depth Buffer 和 Normal Buffer，对每个反射像素：① 计算视线与表面法线的反射方向；② 沿反射方向在屏幕空间做 Ray March；③ 找到与场景深度相交的点；④ 采样该点的颜色作为反射颜色
线性 Ray March（DDA 方法）：
  • 在视图空间沿反射方向等步长（或 NDC 空间 DDA）推进，每步将当前 Ray 位置变换到屏幕空间，比较 Ray 深度与 Depth Buffer 深度
  • 若 ray.z > depth_buffer_sample（Ray 在物体后面），则判定相交，取命中点 UV 采样颜色
  • 缺点：步长固定，空旷区域浪费计算；步长太大会跨越薄物体造成漏检；步长太小则性能差
Hi-Z（Hierarchical-Z）Tracing：
  • 预先构建 Depth Buffer 的 Mip 链（每层保存对应 Tile 的最大深度，即最近深度值），称为 Depth Pyramid
  • Ray March 从高 Mip 层开始（大步长），若 Ray 深度 < 当前 Mip 层最大深度，说明该大 Tile 内肯定无相交，直接跳过（大步前进）；若相交则降至下一 Mip 层精确化
  • 类似 BVH 的层次剪枝：空旷区域迈大步，靠近物体时迈小步，大幅减少 Ray March 迭代次数（通常 10-20 次 vs 线性的 64-128 次）
SSR 完整管线：
  • Pass 1 Ray Tracing：对每个高反射像素做 Hi-Z Ray March，输出命中 UV 和置信度（Fade Factor）
  • Pass 2 Resolve：用命中 UV 采样 Color Buffer，结合粗糙度做模糊（粗糙表面反射模糊）；非命中区域 Fade 到 IBL/Sky 兜底
  • Pass 3 Temporal Denoise：用 Motion Vector 积累历史帧减少噪声，配合 Variance Clamp 抑制 Ghosting
SSR 的主要局限及对策：
  • ❌ 屏幕外几何体无法反射 → 兜底 IBL / Reflection Capture / RT Reflection
  • ❌ 背面几何体不可见 → 对背向摄像机的面不执行 SSR（Normal 朝向判断）
  • ❌ 薄物体穿透 → 引入厚度阈值（Thickness），允许一定的深度误差
  • ❌ 边缘 Fade 明显 → 对屏幕边界和远处渐隐 Fade 到环境贴图，减少突变感
💡 一图串联：SSR = 视线反射方向 → Hi-Z Ray March（大步小步）→ 命中 UV → 采样颜色 → Temporal 降噪。"Hi-Z 相当于用 Depth Pyramid 做 BVH 剪枝"——这个类比能让面试官眼前一亮。

LOD / HLOD / Imposter 系统：原理、适用场景与 UE5 实现
腾讯 字节 高频
▼

📌 参考答案

LOD（Level of Detail）基础：
  • 根据物体与相机的屏幕空间像素面积（Screen Size）动态切换预制的不同精度网格（LOD0 最精细 → LODN 最粗糙），减少远距离无效面数
  • 切换阈值由 ScreenSize 百分比控制；切换时可用 Dithering（Dithered LOD Transition）避免突变，利用 TAA 时域平均实现无缝过渡
  • 代价：美术需为每个 Mesh 手工或自动生成多级 LOD；切换点设置不当会产生 "Popping" 闪跳
HLOD（Hierarchical LOD）：
  • 将场景内多个静态 Mesh 合并成单一的"代理 Mesh"（Proxy Mesh），当整个群组距离足够远时用代理替换所有子物体，大幅减少 DrawCall 数
  • UE5 HLOD 系统支持 World Partition 下的 Runtime HLOD Layer：可配置多层 HLOD，远景自动合并到最粗糙代理；支持 Nanite HLOD（代理也是 Nanite Mesh，无需手动 LOD 链）
  • 适用：开放世界大量重复建筑、树木、石块等静态场景资产
Imposter（公告板 / 广告牌渲染）：
  • 极远距离时，用一张或几张预渲染的 2D 纹理公告板替代 3D Mesh，从任意视角看都近似真实外观
  • Octahedral Imposter：将 Mesh 从八面体均匀分布的方向预渲染成一组图片（含 Albedo/Normal/Depth），运行时根据视角 UV 查表并插值两张相邻视角图像，实现视差修正的伪 3D 效果；适合树木、植被、远景建筑
  • VS Billboarding：传统 Billboard 只有单张正面贴图，无视差修正；Octahedral Imposter 支持完整 360° 视角查找，质量远高于传统 Billboard
选择策略：近景 LOD0（全精度）→ 中景 LOD + HLOD 合批 → 远景 Imposter → 极远 Culling。Nanite 出现后近中景可直接用 Nanite 替代手工 LOD，HLOD 仍负责合并 DrawCall。
💡 面试要点：LOD 解决"面数"，HLOD 解决"DrawCall"，Imposter 解决"极远距离开销"，三者协同覆盖从中景到地平线的全距离范围优化。Nanite 取代了 LOD 链但不能取代 HLOD 的 DrawCall 合并。

Phong 光照模型和 Blinn-Phong 的区别是什么？
腾讯 字节 网易 必考
▼

📌 参考答案

两者都是经验光照模型，由环境光 + 漫反射 + 高光三部分组成。

Phong：高光计算使用 dot(R, V)，R 是反射向量，V 是视线向量
Blinn-Phong：高光计算使用 dot(N, H)，H 是半程向量 H = normalize(L + V)

Blinn-Phong 的优势：
  • 计算半程向量比反射向量更快
  • 当视线与光线夹角大于90°时 Phong 会出现高光截断，Blinn-Phong 不会
  • 结果更接近真实高光分布，能量分布更合理

什么是法线贴图（Normal Map）？TBN 矩阵是什么？
腾讯 网易 高频
▼

📌 参考答案

法线贴图通过纹理存储逐像素的法线方向，模拟表面细节凹凸，而无需增加几何面数。纹理中 RGB 通道对应法线的 XYZ 分量，通常存储在切线空间（Tangent Space），颜色偏蓝表示法线基本朝上。

TBN 矩阵是从切线空间到世界空间的变换矩阵：
  • T（Tangent）：切线，沿 UV 的 U 方向
  • B（Bitangent）：副切线，沿 UV 的 V 方向
  • N（Normal）：几何法线
使用时用 TBN 矩阵将法线贴图中的法线从切线空间变换到世界空间，再参与光照计算。
💡 面试陷阱：法线贴图存储的是切线空间法线，不是世界空间，这是最常考的误区。

如何绘制一个半透明物体？透明排序问题如何解决？
字节 通用 中频
▼

📌 参考答案

半透明物体渲染需要开启 Alpha Blending：srcFactor × src + dstFactor × dst，常用 SrcAlpha, OneMinusSrcAlpha。

透明排序问题：Alpha Blending 依赖绘制顺序，需从远到近渲染（Painter's Algorithm），否则混合结果错误。

解决方案：
  • CPU 端按深度排序后绘制：每帧将透明物体按摄像机距离从远到近排序，依次绘制。适合粒子系统等数量有限、层次简单的场景；缺点是无法处理互相穿插的多边形，排序本身也有 CPU 开销。
  • OIT（Order-Independent Transparency，顺序无关透明）：不依赖绘制顺序，在 GPU 端收集所有片元后统一合成，可正确处理任意穿插透明物体。主要算法见下方专项题。

OIT 方案一：Depth Peeling（深度剥离）的原理是什么？
腾讯 网易 进阶
▼

📌 核心思想

Depth Peeling 通过多轮渲染逐层"剥离"透明层，每轮只取当前最近未处理的透明片元，最后从后往前合成所有层。

📐 算法步骤

  1. 第 0 轮（初始化）：正常渲染所有不透明物体，写入深度缓冲 Z₀（作为背景深度参考）
  2. 第 1 轮（剥第 1 层）：渲染所有透明物体，片元着色器中与 Z₀ 对比，只保留深度 大于 Z₀ 中最小值的片元（即最近的透明层），结果写入颜色纹理 Layer₁ 和新深度 Z₁
  3. 第 N 轮（剥第 N 层）：用上一轮生成的 Zₙ₋₁ 作为参考，重复上述过程，每轮剥出更深一层
  4. 合成：所有层从最远到最近依次 Alpha Blend 合成，得到最终图像

⚙️ 伪代码示意

// 片元着色器核心判断（第 k 轮）
float currentDepth = gl_FragCoord.z;
float prevLayerDepth = texture(prevDepthTex, uv).r;
if (currentDepth <= prevLayerDepth + EPSILON)
    discard;  // 丢弃已经剥过的层
// 通过的片元即为本轮最近未处理层

✅ 优缺点

  • ✅ 精确：结果与排序绘制完全一致，无近似误差
  • ✅ 支持任意穿插的透明物体
  • ❌ 性能差：N 层透明就需要 N+1 次完整渲染 Pass，DrawCall 倍增
  • ❌ 透明层数未知，需预设最大层数或动态判断

改进变种：Dual Depth Peeling — 每轮同时剥最前和最后一层，Pass 数减半，是 Depth Peeling 的工程优化版本。
💡 适用场景：透明层数较少、对正确性要求极高的离线渲染或高端实时渲染（如医学可视化）。游戏中较少直接使用，性能开销过大。

OIT 方案二：Weighted Blended OIT（加权混合透明）原理是什么？
字节 腾讯 进阶
▼

📌 核心思想

Weighted Blended OIT（WBOIT，Morgan McGuire 等 2013 年提出）是一种近似 OIT 方案：不追求逐层精确排序，而是给每个透明片元赋予一个权重（综合考虑深度和不透明度），通过加权平均颜色 + 累积不透明度来近似最终效果，只需 2个 Pass。

📐 算法步骤

  1. Accumulation Pass（累积 Pass）：渲染所有透明物体到两张 RT：
    • accum RT（RGBA16F）：累积加权颜色，accum += color × alpha × w(z, alpha)
    • revealage RT（R8）：累积覆盖率，reveal *= (1 - alpha)
    权重函数 w(z, α) 综合深度和不透明度，示例：w = alpha × clamp(0.03/(1e-5 + (z/200)^4), 1e-2, 3e3)
  2. Composite Pass（合成 Pass）：用全屏四边形合成： 
    vec4 accum = texture(accumTex, uv);
float reveal = texture(revealTex, uv).r;
// 加权平均颜色
vec3 avgColor = accum.rgb / max(accum.a, 1e-5);
// 最终 Alpha
float alpha = 1.0 - reveal;
outColor = vec4(avgColor, alpha);

✅ 优缺点

  • ✅ 性能极好：固定 2 个 Pass，与透明层数无关
  • ✅ 兼容延迟渲染，无需排序
  • ✅ 已被 UE4 / Three.js / Babylon.js 等广泛采用
  • ❌ 近似方案：极端情况（深度接近、高不透明度）效果有偏差
  • ❌ 权重函数需要根据场景调参，不存在通用完美权重
💡 工程首选：WBOIT 是游戏和实时渲染中最常用的 OIT 方案，性价比最高。面试时能说出"2个Pass + 加权平均"即可拿分。

OIT 方案三：Per-Pixel Linked List（逐像素链表）原理是什么？
网易 字节 进阶
▼

📌 核心思想

Per-Pixel Linked List（PPLL）是精确 OIT 方案：利用 GPU 的 Compute Shader + SSBO / UAV，为每个像素维护一条链表，收集该像素上所有透明片元，最后对链表排序后从后往前合成。

📐 算法步骤

  1. 数据结构准备：
    • Head Pointer Buffer：屏幕尺寸的 uint 纹理，每个像素存链表头节点索引，初始化为 NULL（0xFFFFFFFF）
    • Fragment Store Buffer（SSBO）：全局片元节点池，每个节点存 { color, depth, alpha, next }
    • 原子计数器：分配新节点时原子自增，保证并发安全
  2. Fragment Collection Pass（片元收集 Pass）：渲染所有透明物体，片元着色器中： 
    // 原子分配新节点
uint nodeIdx = atomicAdd(counter, 1);
// 写入片元数据
store[nodeIdx] = FragNode(color, depth, alpha,
                           headBuffer[pixelCoord]);
// 头插法更新链表头
imageAtomicExchange(headBuffer, pixelCoord, nodeIdx);
  3. Sort & Resolve Pass（排序合成 Pass）：全屏 Compute Shader，对每个像素的链表按深度从远到近排序（插入排序，片元数通常较少），然后从后往前 Alpha Blend 合成。

✅ 优缺点

  • ✅ 完全精确：结果与 CPU 排序绘制一致，无任何近似
  • ✅ 支持任意数量的透明层，无需预设层数
  • ❌ 显存开销大：Fragment Store Buffer 需预分配，典型值为 8×屏幕像素数个节点
  • ❌ 性能不稳定：透明物体多时显存/带宽压力剧增，原子操作有竞争开销
  • ❌ 需要 D3D11 / OpenGL 4.3+ / Vulkan 才支持

🔄 三种 OIT 方案横向对比

方案 精确度 性能 实现难度 适用场景
Depth Peeling ⭐⭐⭐ 精确 ❌ 慢（N Pass） 中等 离线/医学可视化
Weighted Blended ⭐⭐ 近似 ✅ 快（2 Pass） 简单 游戏/实时渲染首选
Per-Pixel Linked List ⭐⭐⭐ 精确 ⚠️ 中等（显存大） 复杂 高端实时/需精确效果
💡 面试回答策略：先说"OIT有三种主流方案"，再分别用一句话概括各自核心思路，最后说"工程中常用WBOIT因为只需2个Pass且效果可接受"，逻辑清晰，必拿高分。

虚拟纹理（Virtual Texture / Sparse Texture）是如何工作的？Page Table、Physical Tile 和 Indirection Texture 分别起什么作用？
腾讯 网易 中频
▼

📌 参考答案

核心问题：游戏中超大地形、皮肤等材质可能是 16K×16K 甚至更大的纹理，无法完整上传到显存，需要按需加载当前可见的部分。

Virtual Texture（VT）基本架构：
  • 虚拟纹理空间（Virtual Texture Space）：将逻辑上极大的纹理划分为固定大小的 Page（通常 128×128 或 256×256 像素），不实际占用显存
  • Physical Tile Cache：显存中维护一个固定大小的 Tile Pool（如 4096×4096 像素，存储 N 个 Tile），只存放当前帧需要的 Page 数据；当某 Page 被请求且不在 Cache 中时，触发从磁盘/内存的异步流式加载
  • Page Table（页表）：记录每个虚拟 Page 到 Physical Tile 的映射关系，类比 CPU 的虚拟内存页表
  • Indirection Texture（间接纹理）：Page Table 的 GPU 表示，低分辨率纹理（每个 Texel 对应一个 Virtual Page），存储对应 Physical Tile 的位置（UV 偏移 + MIP 级别）
运行时渲染流程：
  • ① Feedback Pass：使用低分辨率 Render Target 渲染一帧，Shader 输出每个像素需要的 Virtual Page ID（虚拟地址 + MIP），由 CPU 读回分析哪些 Page 需要加载
  • ② Page 流式加载：对缺失的 Page 异步从硬盘读取并解压，上传到 Physical Tile Cache 的空闲槽位，更新 Page Table
  • ③ 正式渲染：Fragment Shader 先查询 Indirection Texture 获取物理位置，再在 Tile Cache 中采样真实纹素
  • 备用（Fallback）：Page 未加载时，回退到更粗粒度的 MIP 级别（已常驻显存的低分辨率版本），避免显示黑块
硬件支持：DX11.2+ / Vulkan 的 Sparse Binding（稀疏绑定）允许纹理的内存以 Tile 为单位分别映射，操作系统级别支持虚拟纹理，减少 CPU 拷贝开销。UE5 的 Virtual Shadow Map 和 Nanite 都广泛依赖虚拟纹理思想。
💡 类比记忆：Virtual Texture = 虚拟内存机制用到纹理上。CPU 内存：物理 RAM 是有限的，用 Page Table 把大地址空间映射到小内存；VT：Physical Tile Cache 是有限的，用 Indirection Texture 把大纹理空间映射到小显存。

什么是 PBR？BRDF 的三个核心项 D、F、G 分别代表什么？
腾讯 字节 网易 必考
▼

📌 参考答案

PBR（Physically Based Rendering）基于物理的渲染，核心原则：能量守恒、基于微表面理论、金属/非金属工作流。

Cook-Torrance BRDF：f = D×F×G / (4×(N·L)×(N·V))

  • D（Distribution Function）法线分布函数：描述微表面中朝向半程向量 H 的比例，控制高光大小/形状。常用 GGX（Trowbridge-Reitz）
  • F（Fresnel Equation）菲涅尔方程：描述光在不同入射角下的反射比率，掠射角时反射率趋向1（边缘发光效果）。常用 Schlick 近似
  • G（Geometry Function）几何函数：描述微表面自遮挡和阴影（Self-Shadowing/Masking），常用 Smith GGX
💡 记忆技巧：D 控形状，F 控边缘，G 控自遮。三者缺一不可。

IBL（基于图像的光照）的完整实现流程是什么？预过滤环境贴图与 LUT 如何生成？
腾讯 字节 网易 必考
▼

📌 参考答案

IBL 将 PBR 渲染方程中的光照积分拆分为漫反射和镜面反射两部分预计算，实现实时环境光照。

渲染方程拆分： L_o = L_diffuse + L_specular

1. 漫反射 IBL —— Irradiance Map（辐照度贴图）：
  • 对每个方向 n，对上半球所有入射方向 ωi 积分 L_i(ωi) × cos(θ)
  • 实现：对 HDR 环境贴图进行卷积（通常离线或启动时），生成低分辨率（32×32）的 Irradiance Cubemap
2. 镜面反射 IBL —— Split Sum 近似（Epic Games）：
  • 预过滤环境贴图（Pre-filtered Environment Map）：按粗糙度 [0,1] 生成不同 Mip 层，用重要性采样（GGX 分布）预计算不同粗糙度下的环境反射
  • BRDF LUT（2D 查找表）：预计算 ∫ f_r(n,ωi,ωo) × cos(θi) dωi 关于 (N·V, roughness) 的结果，存为 RG 通道（scale 和 bias），运行时 1 次 textureLOD + 1 次 LUT 采样完成镜面反射计算
实时使用： 
vec3 F = FresnelSchlickRoughness(NdotV, F0, roughness);
vec3 diffuse = irradiance * albedo * (1-F) * (1-metallic);
vec3 prefilteredColor = textureLod(prefilterMap, R, roughness * MAX_MIPS).rgb;
vec2 brdf = texture(brdfLUT, vec2(NdotV, roughness)).rg;
vec3 specular = prefilteredColor * (F * brdf.x + brdf.y);
💡 面试加分：能说出 Split Sum 是 Epic 2013 SIGGRAPH 的近似方案，将积分分为光照项和 BRDF 项分别预计算，是当前游戏引擎 IBL 的工业标准。

什么是次表面散射（SSS）？在实时渲染中有哪些近似实现方案？
腾讯 网易 中频
▼

📌 参考答案

次表面散射（Subsurface Scattering，SSS） 描述光线进入半透明材质内部（如皮肤、玉石、蜡）在内部多次散射后从不同位置射出的现象，使材质看起来柔和、通透而非不透明。

典型现象：强光照射手背时手指泛红光；皮肤在逆光下有半透明的橙红色边缘。

实时近似方案：
  • 纹理空间漫射（Texture-Space Diffusion，Jensen 2001）：在纹理空间对辐照度进行高斯模糊，模拟光在皮下的扩散，多层不同宽度的高斯叠加（d'Eon & Luebke 2007）；UE4 早期皮肤使用此方案
  • 屏幕空间次表面散射（SSSS，Jimenez 2009）：在屏幕空间对漫反射结果进行分离轴高斯模糊，性能更好，是目前游戏中最常用的方案（《神秘海域》《战神》均使用）
  • Pre-Integrated Skin Shading（Penner 2011）：预计算曲率与光照角度的查找表，单 Pass 实现近似 SSS，性能极高，适合移动端
  • 路径追踪 BSSRDF：离线渲染中使用完整 BSSRDF 方程，考虑入射和出射点不同，物理最准确
💡 面试考察点：区分 BRDF（入射和出射同点）与 BSSRDF（入射和出射不同点）的概念差异，是 SSS 物理正确性的核心。

布料渲染为何不能直接用 Cook-Torrance BRDF？Ashikhmin 布料模型和多散射 GGX 各自解决了什么问题？
腾讯 网易 高频
▼

📌 参考答案

Cook-Torrance BRDF 对布料的局限：
  • Cook-Torrance 假设微表面是镜面高光（Specular Microfacets），产生集中的高光光斑
  • 布料由大量纤维构成，纤维横截面是圆柱形，反射呈"逆反射"特性（光从来向反射回去）和"前向前反射"特性，而非集中高光
  • 布料的漫反射更复杂，纤维间多次散射导致光从任意角度均可出射
Ashikhmin 布料 BRDF（用于 Filament/UE 布料着色器）：
  • 使用 倒高斯 NDF（Inverted Gaussian）：D(h) = e^(-tan²θ_h / α²) / (π α² cos⁴θ_h) 的修改形式，高光在掠射角（grazing angle）最强，模拟布料绒毛的"光晕"效果
  • 补充各向异性项，模拟平纹布、丝绸等不同编织结构
  • Filament 引擎中的 Charlie BRDF 对此进一步简化，降低计算成本
多散射 GGX（Multiscattering GGX，Heitz 等，2016）：
  • 标准单散射 GGX 假设光线只与微表面交互一次，在粗糙度高时（α > 0.5）会因丢失多次弹射能量而显得"发暗"，违反能量守恒
  • 多散射 GGX 通过预计算 BSDF Lut（双散射查找表）补偿缺失的多散射能量，使得粗糙材质在高光方向能量守恒
  • UE4/5 的 Energy Conservation 修复 正是基于此，给 Fresnel 反射项乘以能量补偿因子 Ec(f0, roughness)
💡 一句话区分：Ashikhmin/Charlie 布料模型解决的是"布料纤维结构导致的特殊反射形状"问题；多散射 GGX 解决的是"高粗糙度下单散射 BRDF 能量不守恒"问题，两者解决的是不同层次的物理现象。

各向异性材质（Anisotropic BRDF）如何实现？Filament 的多层材质（Clearcoat）模型解决了什么问题？
网易 腾讯 中频
▼

📌 参考答案

各向异性 BRDF（Anisotropic GGX）：
  • 标准 GGX 的法线分布函数（NDF）在各方向上具有相同的粗糙度，高光呈圆形。真实刷拉丝金属、头发等材质在切线/副切线方向粗糙度不同，高光呈椭圆拉伸
  • 各向异性 GGX NDF：引入两个独立粗糙度参数 α_t（切线方向）和 α_b（副切线方向），NDF 公式变为：
    D(h) = 1 / (π·α_t·α_b·[(h·t/α_t)²+(h·b/α_b)²+(h·n)²]²)
  • 实现要求：需要切线向量 T 和副切线向量 B（来自顶点属性或法线贴图中的旋转角 θ），UE5 通过 Tangent Output 节点支持
  • 典型应用：拉丝金属（碳纤维）、唱片盘面、皮肤高光方向拉伸
多层材质（Clearcoat / 清漆层）：
  • 汽车车漆、木地板等材质具有双层结构：底层（有色基底，可粗糙）+ 顶层（透明光滑清漆层）
  • Google Filament 的 Clearcoat 模型：计算两层各自的 BRDF 后线性混合，顶层系数 0~1 可调
  • 能量守恒处理：清漆层的反射会从基础层减去，确保两层加和不超过入射能量：result = clearcoat_brdf + (1 - clearcoat_reflectance) * base_brdf
  • UE5 中对应 Clear Coat 材质模式，需额外提供 Clear Coat Roughness 和 Clear Coat 强度输入
💡 记忆技巧：各向异性 = 把圆形 GGX 高光拉成椭圆，需要切线方向信息；多层材质 = 两个 BRDF 叠加但要保证能量守恒（顶层反射部分不再传入底层）。

VSM（方差阴影贴图）的原理是什么？与 PCF 相比有何优劣？
字节 网易 进阶
▼

📌 参考答案

VSM（Variance Shadow Map） 在 Shadow Map 中额外存储深度的一阶矩（均值 μ）和二阶矩（μ²），利用切比雪夫不等式上界快速估计阴影遮挡概率，无需逐像素多次采样。

核心原理：
  • 存储 (depth, depth²) 到 Shadow Map，可对其进行 任意线性滤波（包括双线性、MipMap、SAT 前缀和）
  • 通过均值和方差重建深度分布，用切比雪夫不等式估计 P(x ≥ t)（被遮挡的概率上界）
  • 不等式：P(x ≥ t) ≤ σ² / (σ² + (t - μ)²)，其中 σ² = μ₂ - μ²

与 PCF 对比：
对比项 PCF VSM
采样次数 多次（开销随核大小增长） 仅 1 次纹理查询
可用滤波器 有限（仅深度比较后求均值） 任意线性滤波（SAT、MipMap）
漏光（Light Bleeding） 无 存在，深度方差大时切比雪夫上界不紧

漏光缓解方法：通过 min/max Clamping 或使用 EVSM（指数方差阴影贴图）可显著减少漏光。
💡 面试加分点：VSM 最大的工程价值在于可以用 SAT（Summed Area Table / Prefix Sum）预计算，使得任意大小软阴影核只需 O(1) 查询，是大核 PCF 的高效替代。

SDF（Signed Distance Field，有向距离场） 存储空间中每个体素到最近表面的有符号距离，负值表示在物体内部。

SDF 阴影原理：
  • 从着色点出发沿光线方向 Ray March，每步根据 SDF 值安全步进（球追踪，Sphere Tracing）
  • 记录光线行进路径上 SDF 值与步进距离的比率最小值，估算阴影遮挡软度（Penumbra Factor）
  • 公式：shadow = min(k × sdf(p) / t)，k 越大阴影越硬

  • 在不支持 RT 硬件的 GPU 上作为 Hardware Ray Tracing 的回退方案
SDF 阴影 vs Shadow Map：SDF 阴影天然支持软阴影且无走样问题，但 SDF 只能近似表示几何（无法处理精细细节），且显存占用大（通常每个 Mesh 一个 3D 纹理）。

Virtual Shadow Map（VSM / UE5）的原理是什么？它如何解决传统 CSM 的问题？
腾讯 字节 进阶
▼

📌 参考答案

Virtual Shadow Map（虚拟阴影贴图，UE5） 是一种超大分辨率（16K×16K 以上）虚拟化的阴影贴图方案，借鉴虚拟纹理（VT）思想，只物理分配实际需要的页（Page）。

传统 CSM（级联阴影贴图）的问题：
  • 固定分辨率无法满足大场景与近距离细节同时高质量的需求
  • 级联边界处阴影跳变，需要 Blend 过渡增加复杂度
  • 动态光源（Lumen 中大量动态光）难以高效管理多套 CSM
VSM 核心机制：
  1. 虚拟化：每个光源的 Shadow Map 定义为超大虚拟纹理（如 16K×16K），逻辑上存在但只有用到的 Page（128×128 像素）才被物理分配到 GPU 显存
  2. Page Table：维护虚拟页到物理页的映射，相机可见区域的像素在光源空间的覆盖范围决定哪些 Page 需要渲染
  3. Cache（持久化缓存）：对静态几何体，已渲染的 Page 缓存起来，只在几何或光源变化时才标记为 Dirty 重新渲染，大幅降低每帧开销
  4. 与 Nanite 配合：Nanite 可直接向 VSM 的 Depth Page 光栅化，无需额外 Shadow Pass
效果：统一的高精度阴影，无级联切换问题，支持大量动态光源，是 UE5 的核心阴影技术。
💡 类比虚拟内存：VSM 与操作系统虚拟内存机制完全类似：虚拟地址空间（超大 Shadow Map）+ 物理内存（GPU 显存 Page Pool）+ 缺页处理（Dirty Page 重渲染）。

PCSS（Percentage Closer Soft Shadows）和 VSSM（Variance Soft Shadow Maps）的软阴影实现原理和性能对比？
腾讯 字节 网易 必考
▼

📌 参考答案

PCSS（2004，NVIDIA）原理：
  1. Blocker Search（遮挡物搜索）：在 Shadow Map 上以当前着色点投影坐标为中心，搜索固定半径内的深度值，计算平均遮挡物深度 d_blocker
  2. 半影宽度估算：根据光源尺寸和遮挡距离估算半影宽度：w_penumbra = (d_receiver - d_blocker) / d_blocker × w_light
  3. 可变半径 PCF：用步骤 2 估算的半影宽度进行 PCF 过滤，半影宽意味着过滤核大，产生模糊软阴影
PCSS 缺点：每次着色需要 2 次大范围采样（Blocker Search + PCF），对于大半影每次可能需要 64+ 次纹理采样，帧率损耗极大。

VSSM（Variance Soft Shadow Maps，2006）原理：
  • Shadow Map 中每个 texel 额外存储深度的均值 μ和方差 σ²（通过 SAT/Mipmap 预计算，O(1) 查询任意区域统计量）
  • 用 切比雪夫不等式（Chebyshev） 近似计算遮挡概率：P(x ≥ t) ≤ σ² / (σ² + (t-μ)²)，O(1) 获得软阴影权重
  • 结合 SAT（Summed Area Table）实现 O(1) 的 Blocker Search 和 PCF 近似，性能极好
VSSM 缺点：漏光（Light Bleeding）问题——当遮挡物深度方差大时，切比雪夫不等式估计过于保守，导致本应在阴影中的区域显示为亮（漏光）。

对比总结：
  • 质量：PCSS > VSSM（PCSS 无漏光）
  • 性能：VSSM > PCSS（VSSM O(1) vs PCSS O(N²)）
  • 工程选择：UE5 VSM 使用 PCSS 类方法配合虚拟页缓存；移动端常用 VSSM 或 EVSM（改进漏光）
💡 EVSM（Exponential VSM） 是 VSSM 的常用改进，用指数变换 e^(c×d) 代替深度值，将深度分布更接近高斯分布，显著缓解漏光问题，是目前移动端软阴影的常用方案。

Cascaded Shadow Map（CSM）的分级策略如何优化？Ray Traced Shadow 与 CSM 相比各有什么优劣？
腾讯 网易 中频
▼

📌 参考答案

CSM 基本原理：
  • 将视锥体沿深度方向划分为 N 个子视锥体（通常 3~4 个），每个子视锥体对应一张独立 Shadow Map（分辨率相同，覆盖范围不同）
  • 近处 Cascade 覆盖小区域（分辨率高），远处 Cascade 覆盖大区域（分辨率低），整体效果比单张 Shadow Map 均匀得多
CSM 优化关键点：
  • 分割策略：对数分割 z_i = z_n × (z_f/z_n)^(i/N) 比均匀分割更符合人眼感知（近处细节更重要）；实际常用对数+均匀的混合分割并加 λ 参数调节
  • Frustum Fitting：将光源正交投影矩阵 Tight-Fit 到子视锥体的 AABB，减少浪费像素；需对 AABB 进行稳定化（Rounding to Texel Size）防止摄像机移动时出现阴影边缘抖动
  • Cascade 混合（Blend Band）：Cascade 边界区域同时采样相邻两级 Shadow Map 并插值，消除边界跳变
  • Shadow Cache / Static Shadow：对不移动的静态物体只渲染一次 Shadow Map 并缓存，动态物体 Dirty 时才局部更新
Ray Traced Shadow vs CSM：
  • CSM 优势：GPU 开销稳定可预期，不依赖硬件 RT，移动端可用
  • CSM 劣势：分辨率有限，远处阴影锯齿明显；无法天然处理大面积光源产生的半影（需额外 PCSS/VSSM）
  • RT Shadow 优势：物理正确的半影，与光源面积精确匹配；支持透明物体和头发自阴影
  • RT Shadow 劣势：需要 DXR 硬件，噪声大（1spp），必须配合 SVGF/NRD 降噪，延迟约 1~2ms/帧（4K 场景）
💡 工程建议：主机/PC 高端游戏常用"CSM 做中远距离阴影 + RT Shadow 做近距离软阴影"的混合方案，兼顾性能和质量。

SSAO、HBAO、GTAO 三代环境光遮蔽算法有何演进？SSGI（屏幕空间全局光照）与 AO 的本质区别是什么？
腾讯 字节 中频
▼

📌 参考答案

环境光遮蔽（AO）的物理含义：场景中某点被周围几何体遮挡时，到达该点的环境光（半球积分）会减少，体现为接缝/褶皱/角落处的暗化效果。

SSAO（Screen Space Ambient Occlusion，Crytek 2007）：
  • 在屏幕空间，对每个像素在其法线朝向半球内随机采样 N 个点（利用 Depth Buffer 重建世界坐标），统计多少采样点深度比当前点更浅（即被遮挡）
  • 缺陷：采样是球形而非半球，需要和法线方向配合才能避免自遮挡；采样点数少时噪声大；遮蔽范围固定，不能表达 大范围低频遮蔽
HBAO（Horizon-Based AO，NVIDIA 2008）：
  • 在多个方向上搜索 地平线角（Horizon Angle），即从当前像素沿某方向在深度图上能"看到"的最大仰角；AO = 1 - sinθ_horizon，物理上更接近真实遮蔽积分
  • 支持法线信息，自遮蔽减少；质量比 SSAO 高，但开销也更大
GTAO（Ground Truth-based AO，Jorge Jimenez 2016，UE5/Unity 默认）：
  • 在 HBAO 基础上引入 精确的 AO 积分公式（基于 Bent Normal 方向的余弦加权积分），并用多重重要性采样估算，结果最接近光线追踪 AO 的"Ground Truth"
  • 引入 Bent Normal（遮蔽方向的等效法线偏转），可用于之后 IBL 采样，使间接光照方向更准确
  • 典型实现：每帧用 4-8 个方向搜索，配合 TAA 时域积累降噪，整体开销约 0.5ms（1080p）
SSGI（Screen Space Global Illumination）与 AO 的本质区别：
  • AO：只计算遮挡因子（0~1），乘以 Ambient Light，不传递有颜色的间接光（颜色渗色 Color Bleeding）
  • SSGI：在屏幕空间采样被遮挡方向的像素 颜色值，将其作为一次反射间接光贡献，能产生 Color Bleeding（如红墙旁白地板变红）
  • 实现思路：类似 SSR，沿法线半球方向做 Ray March，命中的屏幕像素 Radiance 加权积分，再配合 TAA 降噪
  • 代价：只能处理屏幕内可见的遮挡/反射，屏幕外几何体不贡献间接光，且噪声大、需要大量时域积累；UE5 Lumen 在软件光追不可用时回退到 SSGI
💡 演进主线：SSAO（球形采样）→ HBAO（地平线角）→ GTAO（Ground Truth 积分 + Bent Normal）→ SSGI（携带颜色的一次间接反弹）→ Lumen（完整多次弹射全局光照）。每级质量提升伴随计算量递增。

UE5 Virtual Shadow Map（VSM）分页机制深度解析：Page Table / Physical Pool / Clipmap 设计
腾讯 字节 高频
▼

📌 参考答案

VSM 核心动机：传统 Shadow Map 分辨率固定，无法同时满足近处高精度与远处低浪费的需求；CSM 需手动划分 Cascade，过渡边界明显。VSM 借鉴虚拟内存思想，按需分配阴影分辨率。

分页架构（三层结构）：
  • Virtual Page Table（虚拟页表）：逻辑上是一张极高分辨率的 Shadow Map（UE5 中 16K×16K），但实际不分配物理内存；每一个"Page"（如 128×128 texel）对应一个页表项，记录该 Page 是否被分配、在物理池中的地址
  • Physical Page Pool（物理页池）：一块固定大小的纹理 Atlas，存储实际渲染的深度数据；只有被相机可见且在阴影覆盖范围内的 Page 才会被分配到物理池中，节省显存
  • Page 标记（Marking Pass）：每帧先做一次 Marking Pass，遍历屏幕可见像素，计算每个像素对应的阴影 UV，标记哪些 Virtual Page 需要更新；只有被标记的 Page 才在 Shadow Render Pass 中写入 Physical Pool
Clipmap 设计（方向光）：
  • 方向光 VSM 使用类似 Clipmap 的多级 Mip 结构（UE5 默认 8 级），每级覆盖范围是上一级 2 倍，精度降低一半，对应 CSM 的 Cascade 功能但更平滑
  • 相机移动时，Clipmap 以"Page 粒度"滚动更新（类似 Toroidal Addressing），只重新渲染移入视野的新 Page，无需全量重绘，减少每帧开销
Nanite 与非 Nanite 的协同：
  • Nanite Mesh 可直接参与 VSM 的 Rasterize Pass，利用 Nanite 的 Cluster 级裁剪确保只渲染必要三角形
  • 非 Nanite Mesh 有独立的 Marking Job 队列（即常见的"Non-nanite Marking Job Queue overflow"警告来源），当非 Nanite 物体过多或 Queue 容量不足时出现溢出，需增大 CVarShadow.Virtual.NonNaniteMarkingPassBudget
关键优势：稳定高精度（近处可达等效 16K）、无 CSM 接缝、与 Nanite 深度耦合、增量更新减少 Overdraw。代价：Page 管理本身有 CPU/GPU 调度开销，物理池大小是显存瓶颈。
💡 记忆类比：VSM = GPU 虚拟内存。Virtual Page Table = 页表（逻辑地址），Physical Pool = DRAM（物理内存），Marking Pass = 缺页中断检测，Page Update = 缺页补充。Clipmap 滚动 = TLB 刷新的最小化版本。

DLSS / XeSS / FSR 超分辨率的原理是什么？与 TAA 有何关系？
腾讯 字节 高频
▼

📌 参考答案

超分辨率技术以较低分辨率渲染后上采样到目标分辨率，在保证画质的同时大幅提升帧率。

DLSS 2/3（NVIDIA，Deep Learning Super Sampling）：
  • 利用 Tensor Core 运行神经网络，融合当前帧（低分辨率）+ 历史帧（高分辨率）+ Motion Vector 进行时域超分
  • DLSS 3 额外引入 Frame Generation（帧生成）：用 AI 推算相邻两帧之间的插值帧，在 Render 帧率不变的情况下翻倍显示帧率
  • 本质是 TAA 的神经网络强化版，Ghosting 更少，细节恢复更佳
XeSS（Intel，Xe Super Sampling）：
  • 类 DLSS 的时域 AI 超分，在 Intel Arc GPU 上用专用矩阵单元加速，其他 GPU 退化为通用 GEMM（仍有超分效果）
  • 基于 DP4a 指令，兼容范围广
FSR 1/2/3（AMD，FidelityFX Super Resolution）：

UE5 的 TSR（Temporal Super Resolution）与 TAA 的区别是什么？它如何改进 TAA 的 Ghost 和 Blur 问题？
腾讯 字节 高频
▼

📌 参考答案

TAA 的典型问题：
  • Ghosting（残影/重影）：历史帧重投影出错（如遮挡/半透明/新出现物体），导致历史颜色混入当前帧产生拖尾
  • Blur（模糊）：为抑制抖动，历史帧混合比例较高（α ≈ 0.9），移动物体边缘模糊
  • Flickering（闪烁）：亮度对比强烈的细节区域时域不稳定
TSR（UE5 自研，2021）核心改进：
  1. 基于 Velocity 的精准 Rejection：计算逐像素的重投影误差（深度差异 + 颜色差异），对高误差区域的历史帧样本直接拒绝（设权重为 0），不做混合，避免 Ghosting
  2. Subpixel Jitter 的空间重建：利用多帧亚像素偏移样本，在空间域重建超分辨率图像（类似 DLSS 神经网络超分的效果），输出分辨率可超过输入分辨率（如 1080p 输入输出 4K）
  3. Anti-Aliasing Filter：在历史帧合并前对当前帧做空间滤波，统一处理 AA 和超分，不再依赖单独的 MSAA Pass
  4. 移动向量的 Dilation：对深度最近邻的 Motion Vector 做膨胀（Depth Dilation），避免前景物体边缘使用背景运动向量导致的 Ghosting
与 DLSS/FSR 对比：TSR 是纯算法（不依赖神经网络和硬件专用单元），可在所有平台（包括主机）运行；DLSS 需要 Tensor Core，质量更优但依赖 NVIDIA 硬件。
💡 面试要点：TSR 的核心贡献是"基于逐像素误差的 History Rejection"而非简单的颜色 Clamp，这是它比传统 TAA Ghosting 少的根本原因；TSR 同时兼顾了 AA 和超分，是目前跨平台最优的时域算法之一。

  • 神经网络超分（DLSS/FSR 4/XeSS ML）：训练阶段从高分辨率图像学习到超分辨率恢复的先验知识，推理时能根据"已知上下文推断未知细节"，对运动物体、薄线条、高频细节的重建质量高于传统算法
  • 核心权衡：神经网络超分需要专用 AI 硬件（Tensor Core / XMX），延迟相对较高（约 0.5-1ms Tensor Core 时间），但输出质量更接近原始分辨率渲染甚至超越

UE5 TSR（Temporal Super Resolution）的核心原理是什么？与 DLSS 4 / FSR 4 相比有何本质区别？TSR 如何解决时域 Ghosting 和 Rejection 问题？
腾讯 网易 必考
▼

📌 参考答案

TSR 的核心设计目标：
  • UE5 内置的跨平台时域超分方案，无需特定 AI 硬件，在 PC/主机/移动端均可运行
  • 将渲染分辨率降低到 50%（或更低），通过多帧时域积累恢复至 4K 质量，分摊光照计算开销
TSR 核心算法步骤：
  • ① Velocity Buffer（速度场）：TSR 依赖精确的逐像素运动向量（Motion Vector）。静态物体用相机运动矩阵计算，动态物体必须通过 Skeletal/Procedural 动画系统写入 Velocity Buffer；无准确 Motion Vector 的效果（粒子/透明物体）是 TSR 的最大挑战
  • ② History Reprojection：将上帧累积的 History Buffer 用 Velocity 重投影到当前帧像素位置，实现跨帧信息复用
  • ③ Neighborhood Clamping / Variance Clamping：TSR 计算当前帧像素 3×3 邻域的颜色分布（均值+方差），将重投影的历史颜色 Clamp 到邻域包围盒内，抑制鬼影（Ghosting）——这是传统 TAA 也有的操作
  • ④ TSR 专有：Flickering Heuristic（闪烁检测）：在 Clamp 前，TSR 额外检测历史重投影像素是否存在高频闪烁（如金属表面镜面高光跳动），对闪烁区域适度降低历史权重，减少持续性"历史污染"
  • ⑤ Anti-Ghosting Rejection：对运动区域边缘（Disocclusion）或深度不连续处，检测重投影深度与当前深度差异，超过阈值则拒绝历史（Full Rejection），以新帧数据为准，避免边缘残影
  • ⑥ Subpixel Jitter Accumulation：每帧使用 Halton 序列 Jitter 相机，多帧积累不同亚像素位置的样本，等效于在时域做 MSAA，恢复细节
TSR vs DLSS vs FSR 的本质区别：
  • TSR（规则算法）：纯计算机视觉算法，无神经网络，CPU/GPU 通用，质量稳定，不依赖训练数据；Ghosting 和细节丢失仍存在，但明显优于 TAA
  • DLSS 4（NVIDIA 神经网络）：基于 Transformer 的深度学习超分，Tensor Core 加速，利用高分辨率训练数据学习"超分先验"，对薄线、文字、毛发重建质量最高；但仅限 RTX GPU，且每个游戏需专门集成
  • FSR 4（AMD ML 超分）：AMD 推出的机器学习超分（FSR 3 以前是传统算法 EASU+RCAS），FSR 4 引入神经网络，RDNAx 架构加速，开源跨 GPU，但对旧硬件回退到 FSR 3 算法
  • 核心权衡：DLSS（最佳质量，封闭）→ TSR（跨平台，无 AI 硬件依赖）→ FSR（开源通用，质量居中）
💡 面试要点：TSR 的核心在于"Velocity Buffer + History Reprojection + Variance Clamping + Ghosting Rejection"四个串联步骤，每个步骤解决一个具体问题。能说清 TSR 是规则算法而非神经网络、以及与 DLSS 的硬件依赖区别，就覆盖了面试主要考点。

前向渲染与延迟渲染的区别？各自适用场景？
腾讯 字节 网易 必考
▼

前向渲染（Forward Rendering）： 每个物体渲染时直接计算所有光照，复杂度 O(物体 × 灯光)。 优点：简单、支持 MSAA、透明物体友好；缺点：大量灯光时性能急剧下降。

延迟渲染（Deferred Rendering）： 分两步：第一步 Geometry Pass 将几何信息（位置、法线、颜色等）写入 G-Buffer；第二步 Lighting Pass 遍历所有灯光，读取 G-Buffer 计算光照。复杂度 O(像素 × 灯光)。 优点：支持大量动态灯光；缺点：带宽消耗大（G-Buffer）、不支持 MSAA、透明物体需特殊处理。

Forward+（Tile-based）：现代折中方案，将屏幕分 Tile，对每个 Tile 剔除影响它的灯光列表，结合前向渲染，兼顾两者优点。

什么是 SSAO？它的原理是什么？
腾讯 通用 中频
▼

📌 参考答案

SSAO（Screen Space Ambient Occlusion）屏幕空间环境光遮蔽，模拟间接光在角落、缝隙处的遮蔽变暗效果。

原理：
  1. 在每个像素法线半球内随机采样若干点
  2. 将采样点投影到屏幕空间，与深度缓冲中的深度比较
  3. 被遮挡的采样点越多，AO 值越小（越暗）
  4. 对结果进行模糊（Blur）去除噪点
优点：纯屏幕空间，不需要几何预处理；缺点：依赖深度缓冲，存在自遮挡误差，屏幕边缘精度差。

Cluster-Based Deferred Rendering（Clustered Shading）的原理是什么？与传统 Tile-Based 有何区别？
腾讯 字节 进阶
▼

📌 参考答案

Clustered Shading（分簇着色） 是对 Tile-Based Deferred Shading 的三维扩展，将相机视锥体划分为三维网格（Cluster），每个 Cluster 维护影响它的光源列表。

与 Tile-Based 的对比：
维度 Tile-Based（2.5D） Clustered（3D）
空间划分屏幕 2D Tile（如 16×16px）视锥三维 Cluster（XYZ）
深度处理需最小/最大深度，对深度不连续场景效果差深度轴自然分段，适合透明/粒子
光源剔除精度粒度较粗，不考虑深度分布三维剔除，更精确
透明物体难以直接支持天然支持透明物体着色

实现流程：
  1. Cluster 构建：在 Compute Shader 中为每个 Cluster 计算 AABB（视锥空间），Z 轴通常指数划分（近密远稀）
  2. 光源分配：遍历所有点光源/聚光灯，与每个 Cluster AABB 做球-AABB 相交测试，将光源索引写入对应 Cluster 的光源列表
  3. 着色阶段：Fragment Shader 根据片元深度确定所属 Cluster，只遍历该 Cluster 的光源列表进行着色
典型参数：通常划分为 16×9×24 = 3456 个 Cluster，每 Cluster 最多 128 个光源。《战地5》《神秘海域》系列均采用 Clustered Shading 支持数百至数千动态光源。
💡 扩展：Clustered Shading 最初由 Ola Olsson 在 2012 年提出，现已是 AAA 游戏和现代引擎（UE/Unity）处理大规模动态光源的标准方案。

Visibility Buffer（可见性缓冲区）渲染架构是什么？与传统 G-Buffer 延迟渲染相比有何优劣？
字节 腾讯 高频
▼

📌 参考答案

传统 G-Buffer 延迟渲染的问题：
  • 每个可见像素都在 Geometry Pass 中执行完整的顶点变换和材质采样，再写入多张 MRT（GBuffer），带宽开销大
  • 对于使用了大量 Draw Call 的场景，即使最终像素被遮挡，也已消耗了顶点/材质计算资源（OverDraw 问题）
  • G-Buffer 固定格式限制了材质多样性（不同材质需要不同的 G-Buffer 布局）
Visibility Buffer 架构（Burns & Hunt 2013，UE5 Nanite 采用）：
  1. Visibility Pass：仅输出一张 Visibility Buffer（每像素存储 Triangle ID + Instance ID，通常 32/64 bit），只做几何可见性判断，不做任何着色计算
  2. Material Classify Pass：根据 Visibility Buffer 分析哪些屏幕区域覆盖了哪种材质，生成 Material Tile 分类表
  3. Deferred Material Pass：以 Compute Shader 为每种材质生成一个 Full-Screen Pass，只处理该材质覆盖的像素；此时才从 Visibility Buffer 中还原重心坐标，插值顶点属性，采样贴图，计算光照
Visibility Buffer 优势：
  • 极低 Geometry Pass 带宽：Visibility Buffer 仅 4-8 字节/像素，远小于 G-Buffer（通常 16-32 字节/像素 × 3-4 张）
  • 完美支持 Nanite 超密网格：Nanite 通过 Software Rasterizer 直接写 Visibility Buffer，可处理亿级多边形
  • 材质多样性：每种材质独立 Pass，支持任意复杂材质，不受 G-Buffer 格式约束
劣势：
  • 需要在着色阶段手动重建顶点属性（重心坐标插值），实现复杂
  • 透明物体和 Alpha Test 处理更复杂
  • 对大量不同材质的场景，Material Pass 数量多，可能有 Draw Call Overhead
💡 UE5 实现：UE5 Nanite 的核心渲染架构就是 Visibility Buffer，配合 Mesh Shader（或 Software Rasterizer）和 Virtual Shadow Map，实现了"电影级多边形密度 + 实时渲染帧率"的突破。

Forward+、Deferred+ 和 Tile-Based Deferred Rendering 三种架构有什么区别？各自适合什么场景？
腾讯 字节 中频
▼

📌 参考答案

Tile-Based Deferred Rendering（TBDR）：
  • 在传统 Deferred Rendering 基础上，用 Compute Shader 将屏幕划分为 16×16 像素的 Tile，每个 Tile 独立计算其覆盖的光源列表（Light Culling），Fragment Shader 只遍历本 Tile 的光源，而非全部光源
  • 光源剔除：在 Tile 内对光源包围球与 Tile 视锥体做相交测试，通常用深度 Min-Max（从深度缓冲读取）进一步缩小范围
  • 性能收益：对有 1000 个灯光的场景，传统 Deferred 每像素需检查 1000 个灯，TBDR 平均每 Tile 只有几十个灯
Forward+（Tiled Forward）：
  • 保留 Forward Rendering 的渲染流程（支持透明/MSAA/自定义材质），但加入 Light Culling Pass（与 TBDR 相同的 Tile 分组）
  • 在 Forward Pass 中，每个片元只查询本 Tile 的光源列表而非全局灯光
  • 适合场景：需要 MSAA 或复杂材质混合的场景（如 VR/游戏移动端）；UE5 支持 Forward Shading 模式即基于此
Deferred+（Deferred Rendering + Clustered Shading）：
  • 在深度方向上进一步将 Tile 按深度切分为多个 Cluster（三维格子），光源分配到具体 Cluster，解决 TBDR 在深度方向穿插灯光密集时剔除精度不足的问题
  • 适合场景：大开放世界、室内场景（灯光密集且深度方向变化大）；《赛博朋克 2077》夜城使用 Clustered Deferred
三者对比：
Tile Deferred = 屏幕 XY 分块；Forward+ = Tile 分块 + Forward 渲染；Clustered（Deferred+）= XYZ 三维分块，精度最高但内存开销最大。
💡 选型建议：移动端首选 Tile Deferred（ARM Mali 硬件天然 TBDR）；PC/主机复杂场景选 Clustered Deferred；需要 MSAA 或大量透明物体选 Forward+。

OverDraw 指同一像素被多次绘制，浪费 GPU 的着色计算资源，在移动端尤为严重。

LOD（Level of Detail）是什么？有哪些实现方式？
字节 网易 中频
▼

📌 参考答案

LOD 根据物体与摄像机的距离，使用不同精度的模型/纹理，减少远处物体的渲染开销。

实现方式：
  • 静态 LOD：美术制作多套不同面数模型，运行时根据距离切换

Vulkan 同步机制深度解析：Execution Dependency 和 Memory Dependency 有何区别？Pipeline Barrier 的 srcStageMask/dstStageMask/srcAccessMask/dstAccessMask 各自的含义？Image Layout Transition 为何必须配合 Barrier？
腾讯 网易 必考
▼

📌 参考答案

Vulkan 同步的两个维度：
  • Execution Dependency（执行依赖）：保证 A 操作在 B 操作之前完成——即"先做完 A 再做 B"的时序约束。但仅有执行依赖不保证 A 写的数据对 B 可见（Cache 可能未刷新）
  • Memory Dependency（内存依赖）：在执行依赖基础上，强制 A 的写操作从写缓存（Write Cache）刷新到内存，并使 B 能读取的读缓存（Read Cache）失效，从而保证 B 看到 A 最新写入的值。"Execution + Memory Dependency = 完整的数据可见性保障"
Pipeline Barrier 四个关键字段：
  • srcStageMask：指定 Barrier 之前哪些管线阶段的命令必须完成（源阶段）。例如 VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT 表示等待颜色写入阶段完成
  • dstStageMask：指定 Barrier 之后哪些管线阶段的命令被阻塞，直到 Barrier 完成后才能开始（目标阶段）。例如 VK_PIPELINE_STAGE_FRAGMENT_SHADER_BIT 表示后续 Fragment Shader 等待
  • srcAccessMask：指定源阶段中哪种类型的内存访问（写操作）需要被刷新。例如 VK_ACCESS_COLOR_ATTACHMENT_WRITE_BIT 表示颜色写入缓存需要刷出
  • dstAccessMask：指定目标阶段中哪种类型的内存访问（读操作）需要使缓存失效。例如 VK_ACCESS_SHADER_READ_BIT 表示 Shader 读缓存需要无效化
典型用例：GBuffer Pass → Lighting Pass 的 Barrier：
  • srcStageMask = COLOR_ATTACHMENT_OUTPUT（GBuffer 写完颜色）
  • dstStageMask = FRAGMENT_SHADER（Lighting Pass Fragment Shader 要读）
  • srcAccessMask = COLOR_ATTACHMENT_WRITE（刷出写缓存）
  • dstAccessMask = SHADER_READ（使读缓存失效）
Image Layout Transition（图像布局转换）：
  • Vulkan 中 Image 有多种 Layout（UNDEFINED / COLOR_ATTACHMENT_OPTIMAL / SHADER_READ_ONLY_OPTIMAL / PRESENT_SRC_KHR 等），不同操作要求 Image 处于特定 Layout，GPU 内部可能有不同的内部格式/压缩状态
  • Layout 转换必须作为 Memory Barrier 的一部分提交（ImageMemoryBarrier 的 oldLayout → newLayout 字段），与 Execution/Memory Dependency 一并声明，确保转换期间无并发读写
  • 错误示例：跳过 Barrier 直接切 Layout 可能导致验证层报错或 GPU 访问压缩/解压不一致的数据，出现花屏或崩溃
  • UNDEFINED Layout 技巧：若不需要保留 Image 原内容（如每帧 Clear 的 Framebuffer），可用 UNDEFINED → COLOR_ATTACHMENT_OPTIMAL，GPU 可优化跳过实际内容搬运
过度同步的性能陷阱：全局 Barrier（srcStage = ALL_COMMANDS, dstStage = ALL_COMMANDS）会停止整个 GPU 流水线，相当于 GPU 完全 Flush，在移动端开销极大；应精确指定最小化的 Stage 和 Access Mask。
💡 记忆框架：Execution Dependency = 时序保障；Memory Dependency = 数据可见性保障。Barrier 四元组 = (谁写完, 谁等待, 刷出什么缓存, 无效化什么缓存)。Image Layout Transition 只是随 Barrier 顺便声明的状态机切换。

Vulkan Ray Tracing 管线：vkCmdTraceRaysKHR / 着色器绑定表 SBT / RGEN-CHIT-MISS-AHIT 分工
腾讯 高频
▼

📌 参考答案

光线追踪管线 vs 光栅化管线：
  • 光栅化：Draw Call → 顶点着色 → 片元着色，流程固定且按几何体批次执行
  • 光线追踪：vkCmdTraceRaysKHR 发射光线，每条光线独立遍历 BVH 加速结构，命中/未命中时回调不同着色器，流程动态分叉
四类核心着色器：
  • RGEN（Ray Generation）：入口，每个像素一个线程，生成初始射线（方向、起点），调用 traceRayEXT()
  • CHIT（Closest Hit）：光线命中最近几何体时执行，负责着色（采样材质、计算光照、继续发射次级光线）
  • MISS（Miss）：光线未命中任何几何体时执行，负责采样天空盒/环境光照
  • AHIT（Any Hit）：光线穿过半透明几何体时每次相交都回调（可提前终止），常用于透明阴影/Alpha Test
  • Intersection（可选）：自定义基元的相交测试，如程序化球体/曲线
Shader Binding Table（SBT）：
  • SBT 是一块 GPU Buffer，按 四段布局（RayGen Region / Miss Region / Hit Group Region / Callable Region）存储着色器句柄（Shader Handle，32B）和每个几何体/材质实例的用户数据（Shader Record）
  • vkCmdTraceRaysKHR 传入四个 StridedDeviceAddressRegion，GPU 根据光线类型和命中几何体的 Instance/Geometry 偏移自动索引 SBT，找到对应着色器执行
  • 命中组选择公式：HitGroupIndex = RayContribToHitGroupIndex + MultiplierForGeometryContribution × GeometryIndex + InstanceShaderBindingTableRecordOffset，通过这三个字段可为不同材质的不同光线类型（Primary/Shadow/AO）挂载不同的 CHIT/AHIT 着色器
与 DXR 的对应关系：Vulkan SBT = DXR Shader Table；CHIT = DXR ClosestHitShader；MISS = DXR MissShader；vkCmdTraceRaysKHR = DispatchRays。概念一一对应，只是 API 命名不同。
💡 记忆要点：SBT = "光追材质表"，把几何体和着色器动态绑定；RGEN 发射 → BVH 遍历 → CHIT/MISS/AHIT 按情况回调 → RGEN 汇总结果写 Image。SBT 的 Stride 字段允许每个几何体携带不同用户数据（如材质 Index），是光追多材质的关键机制。

光线与三角形的相交检测算法（Möller–Trumbore）？
网易 字节 进阶
▼

📌 参考答案

Möller–Trumbore 算法利用重心坐标求解光线与三角形的交点，不需要先求平面方程。

光线：P = O + t*D，三角形顶点 V0、V1、V2
交点条件：O + t*D = (1-u-v)*V0 + u*V1 + v*V2

用 Cramer 法则求解 [t, u, v]，条件：t>0，u≥0, v≥0, u+v≤1

优点：仅用叉积、点积运算，无需存储法向量，是光线追踪中最常用的求交算法。

GPU 性能分析工具分工：
  • RenderDoc：帧级调试工具，擅长捕获单帧绘制调用（Draw Call）、查看资源（RT/Texture/Buffer）和 Shader 代码，定位"渲染了什么、渲染是否正确"的问题

  • NVIDIA Nsight Systems：系统级时间线视图，适合分析 CPU-GPU 交互（CPU 是否提交不足、GPU 泡沫、CUDA Stream 利用率）

  • Compute-Bound（算力受限）：SM 利用率（SM Active）接近 100%，但 DRAM 带宽利用率较低；主要瓶颈在浮点/整型指令执行；优化方向：减少 ALU 操作、使用 Tensor Core、Mixed Precision
  • Memory-Bound（带宽受限）：SM 利用率不高（Warp 在等待内存），L2 Cache Hit Rate 低，DRAM 带宽利用率高；优化方向：提升缓存命中（改访问模式为 Coalesced）、纹理压缩、减少 KV Cache 带宽（LLM 推理）
  • Latency-Bound（延迟受限）：Occupancy 低（活跃 Warp 数少），Warp 切换无法隐藏延迟；优化方向：提升 Occupancy（减少 Register / Shared Memory 占用）
典型 RenderDoc 分析流程：
  • ① 捕获帧 → Event Browser 找到最耗时的 Pass（通过 Timeline 或 Duration 排序）
  • ② 进入目标 Draw Call → Texture Viewer 验证 GBuffer 正确性
  • ③ Mesh Viewer 检查顶点数、是否有退化三角形
  • ④ Pipeline State 查看 Shader 绑定、RT 格式、Blend State
  • ⑤ 切换 Nsight 做 Shader Profiling：查看 Stall Reason（L1TEX 缺失/数学指令拥塞）

Vulkan 的同步原语（Pipeline Barrier、Semaphore、Fence）各自用于什么场景？Image Layout Transition 是如何工作的？
腾讯 字节 必考
▼

📌 参考答案

Vulkan 显式同步的核心原因：
  • Vulkan 不隐式追踪资源状态，所有执行顺序和内存可见性都必须由开发者声明，这使 GPU 驱动更轻量，但开发负担增加
三种同步原语对比：
  • Pipeline Barrier（管线屏障）：在同一 Command Buffer 内的两个操作之间建立顺序依赖，包含两部分：
    • 执行依赖（Execution Dependency）：指定 srcStageMask → dstStageMask，确保 src 阶段完成后 dst 阶段才开始
    • 内存依赖（Memory Dependency）：srcAccessMask（flush 缓存）→ dstAccessMask（invalidate 缓存），确保写入对后续读取可见
    • 典型用法：Compute Shader 写完 Buffer 后 Barrier 确保 Fragment Shader 能读到最新数据
  • Semaphore（信号量）：用于 Queue 间同步（GPU-GPU），如 Graphics Queue 渲染完成后通知 Present Queue 呈现；或 Transfer Queue 上传完成后通知 Graphics Queue 使用。不能在 CPU 端等待
  • Fence（围栏）：用于 CPU-GPU 同步，CPU 通过 vkWaitForFences 等待 GPU 完成一帧，再重用 Command Buffer 或 Staging Buffer；每帧渲染完成的同步点
Image Layout Transition（图像布局转换）：
  • Vulkan 中 Image 根据用途有不同的 Layout（VK_IMAGE_LAYOUT_UNDEFINED / COLOR_ATTACHMENT_OPTIMAL / SHADER_READ_ONLY_OPTIMAL / TRANSFER_SRC_OPTIMAL 等），不同 Layout 对 GPU 硬件的访问模式优化不同
  • Layout 转换必须通过 Pipeline Barrier 的 ImageMemoryBarrier 完成，指定 oldLayout → newLayout、srcAccess → dstAccess、srcStage → dstStage
  • 常见转换场景：
    • 纹理上传：UNDEFINED → TRANSFER_DST_OPTIMAL（接收数据）→ SHADER_READ_ONLY_OPTIMAL（供 Shader 采样）
    • 渲染到纹理：UNDEFINED → COLOR_ATTACHMENT_OPTIMAL（写 RT）→ SHADER_READ_ONLY_OPTIMAL（后续 Pass 读取）
  • VK_IMAGE_LAYOUT_UNDEFINED：初始状态，内容未定义，转换时 GPU 可以丢弃旧内容（节省带宽，不做 load）
Vulkan Synchronization 2（VK_KHR_synchronization2）扩展：将旧 API 中难以理解的 Stage/Access 标志简化为更语义化的 VkAccessFlags2，并新增 Pipeline Stage2 细粒度控制，是现代 Vulkan 代码的推荐做法。
💡 记忆口诀：Barrier 管 Command Buffer 内部顺序（GPU 内）；Semaphore 管 Queue 之间信号（GPU 间）；Fence 管 CPU 等待 GPU（跨处理器）。Image Layout 记住"先 Barrier 转换，再使用，用完再转出"的三步范式。

什么是 KV Cache？它如何加速自回归生成？
腾讯 字节 高频
▼

📌 参考答案

什么是 MLA（Multi-Head Latent Attention）？它如何减少 KV Cache 开销？
腾讯 字节 进阶
▼

📌 参考答案

MLA（Multi-Head Latent Attention） 是 DeepSeek-V2 提出的注意力改进，核心思想是将 K/V 压缩到一个低维潜在向量（Latent Vector）缓存，显著减少 KV Cache 显存占用。

核心机制：
  • 低秩 KV 压缩：将 K/V 投影到维度远小于原始头维度的潜在空间 c_KV = W_DKV · x，仅缓存 c_KV 而非完整 K/V
  • 推理时解压：每步生成时从 c_KV 解压出完整 K/V 用于 Attention，计算量略增但 KV Cache 大幅缩小
  • Query 侧也压缩：Q 同样先压缩到潜在空间再解压，减少激活显存
与 GQA/MQA 对比：
  • GQA/MQA 通过减少 KV Head 数量来压缩，本质是多个 Q 头共享同一 KV
  • MLA 通过低秩投影压缩，KV 维度降低但每个 token 仍对应独立压缩向量
  • MLA 在相同 KV Cache 预算下，通常比 GQA 获得更好的模型性能（DeepSeek-V2 实验结论）
应用：DeepSeek-V2/V3/R1 均采用 MLA，在 128K+ 上下文场景下相比标准 MHA KV Cache 减少 5~13×。
💡 面试要点："低秩压缩缓存潜在向量而非完整 K/V"是 MLA 与其他方案的本质区别，是 2025 年大模型架构面试的热点。

Mamba（SSM）的原理是什么？与 Transformer 相比有何优缺点？
字节 腾讯 进阶
▼

📌 参考答案

Mamba 是基于 状态空间模型（SSM，State Space Model） 的序列建模架构，是 Transformer 的主要竞争方向。

SSM 核心公式：
  • 连续形式：h'(t) = A·h(t) + B·x(t)，y(t) = C·h(t)
  • 离散化后：通过矩阵 Ā、B̄ 将连续系统转为循环形式，也可展开为卷积计算
Mamba 的关键创新（Selective SSM）：
  • 标准 SSM 的 A/B/C 矩阵是时间不变的，无法选择性关注不同 token
  • Mamba 将 B、C、Δ（步长）设计为输入依赖的（selective），使模型能根据内容动态决定信息压缩比率
  • 通过并行扫描（Parallel Scan）高效训练，推理时退化为 O(1) 状态更新
与 Transformer 对比：
维度 Transformer Mamba
训练复杂度O(n²·d)O(n·d)（并行扫描）
推理（逐步）O(n·d)（KV Cache 线性增长）O(d)（固定状态大小）
长序列建模受限于显存天然支持超长序列
信息召回精确（全局注意力）有损压缩，in-context 召回稍弱

混合架构（Jamba/Zamba）：将 Mamba 层与 Transformer 层交替堆叠，兼顾两者优势，是 2024–2025 年的热门探索方向。

GQA（Grouped Query Attention）和 MLA（Multi-head Latent Attention）分别如何减少 KV Cache 开销？
腾讯 字节 必考
▼

📌 参考答案

KV Cache 的问题：标准 MHA（Multi-Head Attention）推理时每层需缓存所有序列长度的 K/V，显存随序列长度和批大小线性增长，是大模型推理的主要显存瓶颈。

GQA（Grouped Query Attention，Google 2023）：
  • 将 H 个注意力头分为 G 组，每组共享一对 K/V 头（G << H），Q 头仍为 H 个
  • 显存降低：KV Cache 从 O(H) 减少到 O(G)，如 Llama 3（H=32, G=8）节省 75%
  • 质量接近 MHA，明显优于 MQA（Multi-Query Attention，G=1 共享同一对 KV）
  • 已被 Llama 2/3、Mistral、Falcon 等主流开源模型采用
MLA（Multi-head Latent Attention，DeepSeek-V2/V3）：
  • 将 KV 投影到低维潜空间（Low-rank KV Compression），只缓存低维潜向量 cKV 而非完整 K/V
  • 推理时按需从潜向量还原 K/V，KV Cache 大小降低约 5-13×（比 GQA 更激进）
  • 同时引入解耦的旋转位置编码（Decoupled RoPE），保证位置信息准确性
  • DeepSeek-V2 实测：MLA 比 GQA-H8 节省约 93% KV Cache，且在 128K 长上下文下吞吐量大幅提升
💡 2025 趋势：MLA 是 DeepSeek 系列的核心创新之一，已被多个新模型借鉴，面试时能说清楚 GQA 和 MLA 的区别可显著加分。

MoE（Mixture of Experts）模型的工作机制是什么？稀疏激活如何提升参数效率？
字节 腾讯 必考
▼

📌 参考答案

MoE（专家混合模型） 将 Transformer FFN 层替换为多个"专家"网络，每次推理只激活其中少数专家，实现参数量与计算量的解耦。

核心组件：
  • Expert 网络：通常为标准 FFN，总共有 N 个（如 64、128 个）
  • Router（路由器/门控）：对输入 token 的隐藏态做线性变换 + Softmax，选出 Top-K（通常 K=2）分数最高的专家；对应专家输出按 Softmax 分数加权求和
  • 稀疏激活：每个 token 只激活 K 个专家，计算量 ≈ K/N × 全参数量，但模型总参数量巨大（如 DeepSeek-V3 = 671B 总参，37B 激活参数）
训练挑战与解决方案：
  • 负载均衡：Router 可能总是路由到同几个专家导致"热专家"，用 Auxiliary Load Balance Loss 强制均匀分配
  • 通信开销：多 GPU 训练时 Expert 分布在不同 GPU，All-to-All 通信成本高；DeepSeek 引入 Expert Parallelism（EP）优化
  • Expert Capacity：每批 token 中分配给单个 Expert 的最大 token 数，超出则丢弃（Token Drop）或溢出到辅助 Expert
代表模型：Mixtral 8×7B/8×22B、DeepSeek-V2/V3、Switch Transformer、GPT-4（传闻 MoE 架构）。
💡 核心公式：output = Σ(gate_i × Expert_i(x))，其中 gate 由 Router Softmax 的 Top-K 结果决定，其余专家输出乘以 0（稀疏）。

RoPE（旋转位置编码）的原理是什么？相比绝对/相对位置编码有哪些优势？如何支持长上下文外推？
字节 腾讯 必考
▼

📌 参考答案

RoPE（Rotary Position Embedding，苏剑林，2021）核心思想：
  • 不把位置信息加在词向量上（绝对位置编码），而是在计算 Attention 时，对 Q 和 K 的每对 2 维分量按位置索引旋转固定角度
  • 旋转变换：对位置 m 处的向量 x，将其第 2i, 2i+1 维按旋转矩阵旋转 m×θ_i 角度（θ_i = 1/10000^(2i/d)，类似 Sinusoidal）
  • 关键性质：Q_m · K_n = f(x_m, m) · f(x_n, n)，其内积只依赖于相对位置差 (m-n)，天然编码相对位置信息，无需额外的相对位置偏置矩阵
RoPE 的优势：
  • 相对位置感知：Attention 内积隐式包含相对位置，位置感知更自然
  • 外推能力：Sinusoidal 基础使得在训练长度外能稳定旋转，具备一定外推能力（优于 Learned Absolute PE）
  • 高效实现：只需对 Q/K 做逐元素复数乘法（2D 旋转），不增加额外参数，推理计算开销微乎其微
  • 广泛应用：LLaMA、Mistral、Qwen、ChatGLM、DeepSeek 等主流开源模型均使用 RoPE
长上下文外推技术（解决超训练长度退化问题）：
  • 位置插值（PI，Linear Interpolation）：将超出训练长度的位置索引线性缩放到训练范围内，可不微调直接外推，但高频分量信息有损失
  • YaRN（Yet another RoPE extensioN）：对不同频率的旋转分量差异化缩放（低频维度不缩放，高频维度插值），并引入注意力温度修正，大幅减少外推损失；LLaMA-3.1/Qwen2.5 均采用 YaRN
  • LongRoPE：进一步改进旋转基频，允许模型在极长上下文（100万+ token）下稳定运行
💡 面试公式必记：位置 m 的 Q 向量第 (2i, 2i+1) 维经过旋转 [cos(mθ_i), -sin(mθ_i); sin(mθ_i), cos(mθ_i)]，使得 Q_m · K_n 等价于 f(x_m, x_n, m-n)，RoPE 的"相对位置性"正是由此而来。

DPO（Direct Preference Optimization）与 RLHF-PPO 相比有何优势？其数学推导逻辑是什么？
字节 腾讯 高频
▼

📌 参考答案

RLHF-PPO 的复杂性：需要 4 个模型（Actor/Critic/Reward/Reference）同时存在，显存开销大、训练不稳定（PPO 对超参数敏感）、Reward Model 单独训练可能引入误差。

DPO（Direct Preference Optimization，Stanford 2023）的核心思想：
  • 从 RLHF 的最优策略推导出解析解，将 Reward 隐式地参数化为策略本身，不再需要单独的 Reward Model
  • 关键等式：最优策略 π* 满足 r(x,y) = β × log(π*(y|x) / π_ref(y|x)) + β × log Z(x)，其中 Z(x) 为归一化常数，在偏好对差异中消去
DPO Loss： 
L_DPO = -E[log σ(β × (log π_θ(y_w|x)/π_ref(y_w|x)
                          - log π_θ(y_l|x)/π_ref(y_l|x)))]
其中 y_w 为偏好响应，y_l 为不偏好响应，σ 为 Sigmoid 函数。

DPO vs PPO 对比：
  • 模型数量：DPO 仅需 2 个（Policy + Reference），PPO 需 4 个
  • 稳定性：DPO 是监督学习，梯度稳定；PPO 在线强化学习，不稳定
  • 效果：DPO 通常已足够，但 PPO/GRPO 在复杂推理任务（如数学、代码）中更有优势（o1/DeepSeek-R1 均用 GRPO）
变体：SimPO（无 Reference Model）、ORPO（在 SFT 损失中集成偏好对齐）、KTO（不需成对偏好数据）。
💡 2025 现状：DPO 仍是指令微调对齐的主流选择；DeepSeek-R1 用 GRPO（Group Relative Policy Optimization）实现推理能力激发，是当前最热门的 RL 对齐方案。

LoRA 微调的原理是什么？QLoRA 如何进一步降低显存需求？

📌 参考答案

LoRA（Low-Rank Adaptation，2021）：
  • 核心假设：预训练权重的更新矩阵 ΔW 具有低秩结构，可以分解为 ΔW = A × B，其中 A ∈ Rd×r，B ∈ Rr×k，r << min(d,k)
  • 前向传播：h = Wx + (BA)x × (α/r)，α 为缩放系数
  • 参数节省：只训练 A、B 两个小矩阵（参数量约为全参的 0.1%~1%），原始权重 W 冻结
  • 推理无额外延迟：可以将 A×B 合并回 W，推理时与全量权重等价
QLoRA（2023，UW）：在 LoRA 基础上进一步降低显存：
  • NF4（4-bit NormalFloat）量化：将预训练权重量化为 4-bit，专为正态分布权重设计，精度损失极小
  • Double Quantization（双重量化）：对量化参数（scale）再次量化，进一步节省显存
  • Paged Optimizer：利用 CPU 内存作为 GPU 显存溢出缓冲，避免 OOM
  • 效果：65B 模型可在 48GB 显存（单卡 A100）上微调，而全量 BF16 需要约 780GB
常用变体：DoRA（Weight-Decomposed LoRA，分别 LoRA 幅度和方向）、LoRA+（差异学习率 A/B）、MoLoRA（多 LoRA 专家混合）。
💡 面试公式必记：h = Wx + BAx × (α/r)，能白板手推且解释各字母含义是加分项。r 通常取 8/16/64，越大效果越接近全量微调。

GRPO（Group Relative Policy Optimization）的原理是什么？DeepSeek-R1 如何用它激发模型的推理链（Chain-of-Thought）能力？
字节 腾讯 必考
▼

📌 参考答案

PPO 的问题：PPO 需要训练一个 Critic Value 网络（大小与 Actor 相同）来估计基线（Baseline），显存开销是 Actor 的 2 倍，且 Critic 训练不稳定。

GRPO（DeepSeek，2024）核心思想：
  • 组内相对优势：对同一道问题（Prompt），用当前策略采样 G 个回答（如 G=8），用规则 Reward（如数学答案正确 +1/错误 -1，格式 +0.5）对每个回答打分 r_i
  • 无需 Critic：以组内所有回答的平均分作为基线：A_i = (r_i - mean(r)) / std(r)，直接计算相对优势，省去 Critic Value 网络
  • KL 散度约束：同 PPO 一样加入 KL 惩罚项，防止策略偏离参考模型太远
GRPO Loss： 
L = -E[min(r_t × A_t, clip(r_t, 1-ε, 1+ε) × A_t)] + β × KL(π_θ || π_ref)
其中 r_t = π_θ(a_t|s_t) / π_old(a_t|s_t) 为重要性比。

DeepSeek-R1 的训练流程（冷启动 → RL → 蒸馏）：
  1. 冷启动 SFT：用少量高质量 CoT 数据（含 <think> 标签）做监督微调，给模型基础推理格式
  2. GRPO RL 训练：以数学、代码等有明确答案的题目为训练集，奖励函数 = 结果正确奖励 + 格式遵循奖励（是否有 <think> 结构），GRPO 驱动模型自主涌现出长 CoT 推理行为（Aha Moment）
  3. 拒绝采样 + SFT：用 R1 生成的高质量 CoT 数据过滤后做第二轮 SFT
  4. 第二轮 GRPO：进一步对齐多场景
  5. 蒸馏：用 R1 生成的推理链数据蒸馏到 1.5B-70B 小模型，即 R1-Distill 系列
💡 关键洞察：GRPO 的优雅之处在于"用组内相对分数替代绝对价值估计"，消除了 Critic 网络，将 RL 训练从 4 个模型降为 2 个（Actor + Reference），是使推理模型训练工程可行的核心简化。

FlashAttention 的核心优化思路是什么？FlashAttention-2/3 做了哪些改进？
字节 腾讯 必考
▼

📌 参考答案

标准 Attention 的瓶颈：计算 softmax(QKT/√d) × V 时，S=QKT 矩阵大小为 N×N（N 为序列长度），需要写回 HBM（显存），是典型的 Memory Bound 操作，对 N=4096 就需要 64GB 级的 HBM 读写。

FlashAttention（Dao 等，2022）核心思路：
  1. 分块（Tiling）：将 Q/K/V 切分为适合 SRAM 的小块（如 SRAM 容量 ~192KB），每次只在 SRAM 内计算一个子块的 Attention，避免将中间结果 S/P 写回 HBM
  2. 在线 Softmax（Online Softmax）：利用 Softmax 的分块可重计算性（维护 running max 和 running sum），无需在 HBM 中存储完整 N×N 矩阵
  3. 重计算（Recomputation）：反向传播时不存储前向的 Attention 矩阵，而是重新计算，用计算换显存
复杂度：HBM IO 从 O(N²) 降为 O(N)，计算复杂度仍 O(N²d) 不变；实测训练速度提升 2-4×，显存减少 5-20×。

改进版本：
  • FlashAttention-2（2023）：优化线程分配（减少 non-matmul FLOP），更好利用 Tensor Core，进一步提升 GPU 利用率（A100 可达 70%+ FLOP 利用率）
  • FlashAttention-3（2024）：针对 H100 Hopper 架构，异步 WGMMA + 流水线化，同时优化 FP8 支持，吞吐接近 H100 理论峰值的 75%
💡 一句话总结：FlashAttention 不是改变 Attention 的数学等价结果，而是通过 IO 感知算法（IO-Aware Algorithm）减少 HBM 读写，将 Attention 从 Memory Bound 变为 Compute Bound。

投机采样（Speculative Decoding）的原理是什么？它如何在不改变输出分布的情况下加速推理？
字节 腾讯 高频
▼

📌 参考答案

LLM 推理的瓶颈：自回归解码每次只生成 1 个 token，每次前向传播都需要加载全部模型权重（Memory Bandwidth Bound），GPU 利用率极低（通常 <20%）。

投机采样（Speculative Decoding，Google 2023）核心思路：
  1. Draft Model 起草：用一个小模型（Draft Model，如 68M 或 7B）自回归生成 K 个 token（draft tokens）
  2. Target Model 并行验证：将 K 个 draft tokens 一次性送入大模型（Target Model）并行前向，得到每个位置的概率分布
  3. 拒绝采样（Rejection Sampling）：按接受概率 min(1, p_target(t_i) / p_draft(t_i)) 逐个验证；全部接受则额外采样 1 个 Target Model token；遇到拒绝则截断，从拒绝位置重新用 Target Model 采样，输出分布与纯 Target Model 完全相同（无偏）
加速原理：
  • 大模型并行验证 K 个 token 比顺序生成 K 个 token 快（相同 KV Cache 大小，更好利用 Tensor Core）
  • 平均接受长度 α 越高（draft 越准），加速比越大；实测 2-3× 加速，输出质量等价
变体：
  • Self-Speculative Decoding：用同一模型的早期层作为 Draft（减少 Draft Model 依赖）
  • Medusa：在模型顶部添加多个 Draft Head，并行预测多个位置
  • EAGLE（字节，2024）：用自回归 Draft Head + 特征拼接，Draft 准确率显著提升，加速 3× 以上
💡 关键保证："输出分布与 Target Model 等价"是投机采样的核心性质，通过拒绝采样的数学修正实现，面试时一定要强调这一点而非说"有近似"。

MHA/MQA/GQA/MLA 四种 KV Cache 方案的区别是什么？DeepSeek-V2 的 MLA 如何在低 KV Cache 下保持高质量？
字节 腾讯 必考
▼

📌 参考答案

背景：自回归推理时，每个 token 生成都需要所有之前 token 的 K/V 向量（KV Cache），长序列下 KV Cache 大小 = 2 × num_layers × seq_len × num_heads × head_dim，是推理显存的主要瓶颈。

四种方案对比：
  • MHA（Multi-Head Attention）：每个 Head 独立存储 K/V，KV Cache = H × d（H = 头数，d = 每头维度），质量最高但开销最大；GPT-2/BERT 使用
  • MQA（Multi-Query Attention，Shazeer 2019）：所有 Query Head 共享同一组 K/V（只有 1 个 K Head 和 1 个 V Head），KV Cache = 1/H 倍，推理带宽极省，但质量略降；PaLM、Falcon 使用
  • GQA（Grouped-Query Attention，Google 2023）：将 H 个 Query Head 分成 G 组，每组共享 1 对 K/V，KV Cache = G/H 倍（G 通常取 H/8 或 H/4），质量接近 MHA；LLaMA-2 70B、Mistral、Qwen2 等广泛使用
  • MLA（Multi-head Latent Attention，DeepSeek-V2 2024）：将 K/V 压缩为低维潜变量 c（通过 Down-Projection 矩阵），推理时只缓存压缩后的 c；计算注意力时用 Up-Projection 还原 K/V。KV Cache 压缩到 MHA 的约 1/9，且质量接近甚至超过 MHA
MLA 的核心设计：
  • 训练时：K = W_UK × c_KV，V = W_UV × c_KV，c_KV = W_DKV × h（Down-Projection，d → d_c，d_c << d）
  • 推理时：只缓存 c_KV（低维），不缓存完整 K/V；但计算 Attention 需要还原，通过将 W_UK 合并到 Q（吸收技巧），可在不还原 K/V 的情况下直接计算注意力得分（节省还原开销）
  • 额外引入 RoPE 解耦：MLA 对 Q/K 的旋转位置编码部分单独处理（Decoupled RoPE），与潜变量路径分离
实测效果（DeepSeek-V2，236B MoE 模型）：KV Cache 降低到 MHA 的 5.75%，推理吞吐提升 5.76×，质量持平或优于 MHA。
💡 记忆顺序：MHA（全缓存）→ MQA（共享1组）→ GQA（分组共享，折中）→ MLA（压缩潜变量，最激进），这条进化链是推理优化显存的核心路线，面试时能完整说出是高分项。

GraphRAG 与传统向量 RAG 的区别是什么？它如何解决多跳推理问题？
字节 腾讯 中频
▼

📌 参考答案

传统向量 RAG 的局限：
  • 每次检索独立，无法利用文档间的关联关系
  • 多跳问题（如"A 的老板的公司在哪里？"）需要跨多个文档推理，向量相似度检索难以捕捉
  • 全局总结性问题（如"整个数据库的主要主题是什么？"）无法通过局部检索解答
GraphRAG（Microsoft，2024）核心流程：
  1. 知识图谱构建：用 LLM 从文档中抽取实体（Entity）和关系（Relation），构建 KG（知识图谱）
  2. 社区检测：用图算法（如 Leiden 算法）将 KG 划分为层次化社区，对每个社区生成摘要
  3. 查询处理：
    • Local Search：针对特定实体的问题，从 KG 中检索相关实体邻域 + 对应文本
    • Global Search：针对全局问题，汇聚社区摘要，多层次 Map-Reduce 生成答案
多跳推理优势：KG 中实体间的关系路径天然支持多跳游走（如 A -[老板]→ B -[公司]→ C），检索时可沿关系链扩展，不受向量相似度检索的"单跳"限制。

局限：构建 KG 需要大量 LLM 调用（成本高），图的质量依赖抽取准确性，实时更新困难。
💡 适用场景：GraphRAG 适合文档间关联强、需要全局摘要或多跳推理的场景（如企业知识库、法律文书、学术论文）；传统 RAG 适合事实性单次检索场景。

HyDE（假设文档嵌入）和混合检索（Hybrid Search）如何提升 RAG 的检索质量？Reranker 的作用是什么？
字节 腾讯 中频
▼

📌 参考答案

基础 RAG 的检索质量问题：
  • 语义鸿沟：用户 Query 短而模糊（如"如何减少推理延迟"），与文档段落语义空间差异大，向量相似度低
  • 关键词遗漏：纯向量检索无法精确匹配专有名词、代码符号等 Sparse 特征
  • Recall vs Precision 矛盾：检索 Top-K 大则召回高但噪声多；K 小则精确但可能遗漏关键文档
HyDE（Hypothetical Document Embeddings，2022）：
  • 核心思想：先用 LLM 根据 Query 生成一段"假设性答案文档"（Hypothetical Document），再用这段假设文档的向量去检索真实文档库
  • 优势：假设文档与真实知识文档的语义空间更接近（都是文档风格），检索准确率显著提升，尤其对专业领域问题效果突出
  • 代价：多一次 LLM 调用（延迟增加），假设文档质量依赖 LLM 能力
混合检索（Hybrid Search）：
  • 稠密检索（Dense Retrieval）：向量相似度搜索（如 FAISS、Milvus），适合语义理解
  • 稀疏检索（Sparse Retrieval）：BM25/TF-IDF 关键词匹配，适合精确专有名词匹配
  • 融合方式：RRF（Reciprocal Rank Fusion）对两路检索结果按排名倒数求和重新排序：score = Σ 1/(k + rank_i)，k 通常取 60
Reranker（精排）：
  • 检索阶段（Recall）速度优先，召回 Top-50~100 个候选文档
  • Reranker（如 BGE-Reranker、Cohere Rerank）对 Query-Document 对做交叉注意力精打分（Cross-Encoder），精排到 Top-5~10
  • Cross-Encoder 比 Bi-Encoder（向量检索）精度高但速度慢（不能预计算文档向量），是精排-召回两阶段的完美配合
完整高质量 RAG Pipeline：Query → HyDE 假设文档 → 混合检索（Dense + Sparse，RRF 融合）→ Reranker 精排 → Top-K 文档 → LLM 生成答案。
💡 面试加分点：能描述"Recall 阶段用 Bi-Encoder（速度快），Rerank 阶段用 Cross-Encoder（精度高）"的两阶段架构，并解释为什么不能直接用 Cross-Encoder 做全库检索（复杂度 O(N)，N 为文档数，无法预计算向量）。

  • Dynamic Resolution：将高分辨率图像切分为多个 448×448 的 Tile，每个 Tile 独立编码后拼接，支持超高分辨率输入（OCR、文档理解）
  • InternVL2 在多项 VQA/OCR 基准超越 GPT-4V
多模态训练三阶段：
  • ① 图文对齐预训练（只训练 Projector，冻结 ViT 和 LLM）
  • ② 视觉指令微调（解冻 LLM，训练 Projector + LLM）
  • ③ 偏好对齐（RLHF / DPO，减少幻觉）
💡 面试关键点：强调"视觉 Token 数量 = 图像 Patch 数 = 推理时 KV Cache 开销的主要来源"，高分辨率图像会产生 1000+ 视觉 Token，这是 VLM 推理优化（Token 压缩）的核心动力。

扩散模型（Diffusion Model）的数学原理是什么？DDPM、DDIM 和 Flow Matching 各有什么特点？
字节 腾讯 中频
▼

📌 参考答案

DDPM（Denoising Diffusion Probabilistic Models，2020 Ho et al.）：
  • 前向过程（加噪）：逐步向真实数据 x₀ 添加高斯噪声，经 T 步后得到纯噪声 xT，满足马尔可夫链：q(x_t | x_{t-1}) = N(x_t; √(1-β_t)·x_{t-1}, β_t·I)
  • 逆向过程（去噪）：神经网络（U-Net）学习预测每步的噪声 ε，从 xT 开始逐步去噪还原 x₀
  • 训练目标：MSE(ε_θ(x_t, t) - ε)，即预测加入的噪声
  • 缺点：逆向推理需要 1000 步，采样极慢
DDIM（Denoising Diffusion Implicit Models，2021 Song et al.）：
  • 将 DDPM 的马尔可夫链改为非马尔可夫过程，可跳步采样（如只走 50 步），大幅加速推理
  • DDIM 采样具有确定性（给定同一噪声，输出确定），支持图像插值
  • 代价：同等步数下质量略低于 DDPM，但 50 步 DDIM ≈ 1000 步 DDPM 的大部分质量
Flow Matching（Lipman et al. 2022；用于 Stable Diffusion 3、FLUX）：
  • 不走加噪/去噪的曲折路径，而是学习数据分布到噪声分布之间的直接流场（Vector Field），路径更直，推理步数可进一步减少到 8~20 步
  • Rectified Flow：将随机数据-噪声配对的路径"拉直"，减少积分误差，是 SD3/FLUX 的核心技术
  • 训练目标：预测给定时刻 t 的速度向量 v = x₁ - x₀（从噪声到数据的方向）
💡 演进脉络：DDPM（慢但准）→ DDIM（加速采样）→ Flow Matching（路径更直，更快更稳）→ Consistency Model（1步采样，蒸馏方向）。当前 AIGC 图像生成模型（SD3、FLUX.1、Sora）基本都基于 Flow Matching 框架。

模型量化（GPTQ、AWQ、SmoothQuant）的原理是什么？INT4/INT8 量化对推理性能和精度的影响如何权衡？
字节 腾讯 必考
▼

📌 参考答案

量化核心目标：将 FP16/BF16 权重压缩为 INT8 或 INT4，减少显存占用（4× 或 8×）和内存带宽，提升推理 token/s。

GPTQ（PTQ，2022 Frantar et al.）：
  • 基于 OBQ（Optimal Brain Quantization）框架，逐层、逐行量化权重矩阵
  • 利用 Hessian 矩阵信息（通过校准数据集计算）补偿量化误差：量化一个权重时，同步调整同行其他权重抵消误差
  • 特点：纯权重量化（Weight-Only），激活值仍为 FP16；无需重训练，量化速度快（单卡数小时）；INT4 W4A16 格式是当前 LLM 部署主流之一
AWQ（Activation-aware Weight Quantization，2023 MIT）：
  • 观察到权重中约 1% 的"显著权重"（对应激活值大的通道）对量化误差贡献最大
  • 对显著权重所在通道进行缩放（Per-channel Scale）后再量化，等效于在激活空间做保护，减少精度损失
  • 特点：比 GPTQ 在低比特（W4）下精度更高；推理时无需特殊 Kernel，兼容性好；已集成进 vLLM 和 TGI
SmoothQuant（W8A8，2022 MIT + NVIDIA）：
  • 激活值量化（A8）比权重量化更难，因为激活值有"异常值（Outlier）"：极少数通道值远大于其他通道，导致量化误差爆炸
  • 将激活值的缩放难度通过数学等价转移给权重（Per-channel 缩放因子 s，激活除 s，权重乘 s），使激活和权重都"平滑"，同时量化精度高
  • 支持 W8A8 推理，可利用 TensorCore INT8 计算单元，吞吐量比 W4A16 更高（算力受限场景优势明显）
精度影响：INT8 ≈ FP16 精度损失极小（<1% PPL）；INT4 W4A16 PPL 约上升 2~5%，但 Llama-3 70B INT4 仍可超越 Llama-3 8B FP16，是性价比最优解。
💡 选型建议：显存极限（单卡部署 70B）首选 AWQ W4；高吞吐批量推理（算力密集）选 SmoothQuant W8A8；快速实验选 GPTQ（工具链最成熟）。

Prefill-Decode 分离推理架构是什么？PD 分离如何解决 LLM 推理的吞吐量瓶颈？
字节 腾讯 必考
▼

📌 参考答案

LLM 推理的两阶段特征：
  • Prefill 阶段：处理用户输入 Prompt（并行计算所有 Token 的 KV Cache），是计算密集型（Compute-bound）：大矩阵乘法，GPU 利用率高，单次耗时随 Prompt 长度线性增长
  • Decode 阶段：每次只生成 1 个新 Token（Autoregressive），是内存带宽密集型（Memory-bound）：每步需要加载全部 KV Cache，GPU 计算单元大量空闲
混合部署的问题：
  • 在同一 GPU 上同时运行 Prefill 和 Decode 时，两者互相干扰：Prefill 占用大量计算资源导致 Decode 延迟增加（Time-to-First-Token 变长）；Decode 频繁插入导致 Prefill 吞吐下降
PD 分离（Disaggregation）架构：
  • 核心思想：将 Prefill 和 Decode 部署在不同的 GPU 实例上，各自独立扩展
  • Prefill 机器：高计算密度，可用 H100/A100 满载 Prefill 请求，批量处理，计算 GPU 利用率 > 90%
  • Decode 机器：更多内存带宽，专注生成 Token，可用 H100 NVLink 或 AMD MI300X（大 HBM）提升 KV Cache 吞吐
  • KV Cache 在 Prefill 完成后通过高速互联（NVLink/InfiniBand）传输到 Decode 节点
典型实现：
  • 字节跳动 Mooncake（2024）：基于 PD 分离的 KV Cache 调度系统，TTFT 降低 50%+
  • NVIDIA TensorRT-LLM disaggregated serving（2024）
  • SGLang / vLLM v0.6+：支持 chunked prefill 和 disaggregated prefill
代价：需要额外的 KV Cache 传输开销和更复杂的调度逻辑，适合长 Prompt + 长输出的业务场景（如 Code Generation、Document Analysis）。
💡 2025 趋势：PD 分离已成为大规模 LLM 推理集群的标准架构。面试可提及"Prefill is compute-bound, Decode is memory-bandwidth-bound"这一核心洞察，再引出 PD 分离的动机。

RoPE 位置编码为何在长上下文外推时失效？YaRN 和 LongRoPE 各用什么方法扩展上下文窗口？
字节 腾讯 必考
▼

📌 参考答案

RoPE（旋转位置编码）基础：
  • RoPE 对 Q/K 向量的每对维度以位置相关的旋转矩阵编码，实现相对位置感知：q_m · k_n = f(m-n)
  • 不同维度对应不同频率（低维度高频，高维度低频），类比傅里叶分解
长上下文外推失效原因：
  • 训练时最大序列长度为 L（如 4096），高频维度对应的旋转角度在 L 以内有"周期覆盖"，而超过 L 后，高频维度旋转角度进入训练从未见过的区域，注意力分数分布异常（即"OOD 位置"），模型输出质量急剧下降
  • 位置 m > L 时，θ_i × m 超出训练分布，模型无法正确计算远距离相对位置关系
线性插值（Position Interpolation，PI）：将位置 m 等比缩放为 m × (L_train / L_target)，把超出训练范围的位置压回到 [0, L_train] 内，再微调（Fine-tune）几百步即可；代价是近距离位置分辨率降低。

YaRN（2023，Mistral 等采用）：
  • 观察到不同频率维度的 OOD 程度不同：低频维度外推时表现良好（旋转角变化慢），高频维度外推时失效严重
  • 分组处理：低频维度 → 不缩放；高频维度 → 线性插值缩放；中频维度 → 混合策略（NTK-aware）
  • 加入 Temperature 缩放（attention 温度系数），修正插值后注意力熵的变化，微调步骤远少于纯 PI
LongRoPE（2024，Microsoft）：
  • 对每个 RoPE 维度搜索最优的非均匀缩放因子（通过进化算法搜索），而非手工设计分组规则
  • 两阶段策略：先在 256K 上训练，再用调整后的 RoPE 缩放扩展到 2M Token
  • 在 Phi-3 Mini（3.8B）上实现 128K 上下文，性能接近 4K 原始窗口
主流模型实践：Llama 3 用 8:1 的 Scaling Factor + YaRN 扩展到 128K；DeepSeek-V3 使用 YaRN 扩展到 128K；Gemini 1.5 用环形注意力（Ring Attention）实现 1M Token。
💡 记忆框架：RoPE 外推失效 → 根本原因是高频维度 OOD → PI（简单线性压缩）→ NTK-aware（频率分组差异化处理）→ YaRN（加温度修正）→ LongRoPE（维度级最优搜索）。越后越精细但越复杂。

视频生成模型（Sora/DiT）的架构原理是什么？相比图像扩散模型，视频生成在时空建模上做了哪些关键扩展？
字节 腾讯 中频
▼

📌 参考答案

DiT（Diffusion Transformer，Peebles & Xie 2023）：
  • 将扩散模型的去噪 U-Net 骨干替换为 Transformer（ViT 风格），在 Latent Space 中操作（先用 VAE 将图像压缩到 Latent）
  • 每个 DiT Block：LayerNorm → Self-Attention → Cross-Attention（注入条件如文本嵌入）→ Feed-Forward，时间步 t 通过 AdaLN-Zero（自适应层归一化）注入
  • Scaling Law 表现优秀：模型越大（DiT-XL）生成质量越高，且无 U-Net 的跳连接复杂度
Sora（2024，OpenAI）的关键扩展：
  • 时空 Patch（Spacetime Patch）：将视频的时间帧和空间 Patch 统一为 3D Patch Token（t × h × w），Transformer 同时在时间和空间维度上做 Self-Attention，天然处理任意分辨率和时长
  • 3D VAE 压缩：用 3D 卷积 VAE 在时空维度同时压缩视频（时间维度约 4× 压缩，空间维度 8×），大幅减少 Latent Token 数量，使 Transformer 处理长视频可行
  • 文本条件注入：T5 / CLIP 文本编码器输出 Cross-Attention Key/Value，Transformer 每层通过 Cross-Attention 注入语义条件
视频生成的核心挑战：
  • 时序一致性：相邻帧之间物理运动需要连贯，纯 2D 扩散（逐帧独立生成）会抖动；需要时间维度的 Attention 或 3D 卷积显式建模帧间关系
  • 计算量爆炸：32 帧 1080p 视频 = 数千个 Spatial Patch × 32 帧，Self-Attention 复杂度 O(N²)，需要稀疏 Attention（Factorized Spatial-Temporal Attention：先做 Spatial，再做 Temporal）
  • 长视频生成：超过 10 秒视频的语义连贯性（物体持续存在、背景一致），目前主流方法是滑动窗口 + 条件帧作为 Anchor
国内主要视频生成模型：字节 MagicVideo-V2 / Wan Video（万象）；智谱 CogVideo；快手 Kling；阶跃星辰 Step-Video——均以 DiT + 3D VAE 为基础架构。
💡 面试要点：视频生成 = 图像扩散 + "时间维度建模"，重点说清楚：① DiT 替代 U-Net 的优势；② 3D Patch Tokenization 如何统一处理视频；③ 时空注意力如何权衡计算量与时序一致性。

LLM 知识蒸馏与结构化剪枝的原理是什么？DeepSeek-R1 对小模型蒸馏的方式和传统 KD 有何不同？
字节 腾讯 必考
▼

📌 参考答案

知识蒸馏（Knowledge Distillation，KD）基础：
  • 核心思想：用大 Teacher 模型的"软标签"（Soft Label / Logit）监督小 Student 模型训练，比用 One-hot 硬标签包含更丰富的类间相似性信息
  • 损失函数：L = α × CE(student, hard_label) + (1-α) × KL(softmax(student_logits/T), softmax(teacher_logits/T))，温度系数 T 用于软化分布（T 越大，分布越平滑，传递更多"暗知识"）
  • 中间层蒸馏（Feature Distillation）：除 Logit 外，还可对 Teacher 的隐藏层激活、Attention Map 进行 MSE 匹配，效果更好但需要维度对齐
结构化剪枝（Structured Pruning）：
  • 非结构化剪枝：将权重矩阵中绝对值小的参数置零（稀疏），但稀疏矩阵在 GPU 上实际加速有限（需 Sparse Tensor Core 支持）
  • 结构化剪枝：整体移除 Attention Head、FFN Intermediate 维度、甚至整个 Transformer Layer，输出仍是密集矩阵，标准 GPU 可直接加速
  • LLM-Pruner（2023）：基于梯度重要性评估，对 Coupled Structure（依赖关系组）批量剪枝，保持模型功能完整性
  • NVIDIA 的做法（Llama-3.1 8B → 4B）：先深度剪枝（宽度减半）→ 再用教师模型持续蒸馏（数十亿 Token 训练），剪掉 50% 参数但保留 ~95% 的性能
DeepSeek-R1 的蒸馏方式（2025）：
  • 与传统 KD 的关键区别：DeepSeek-R1 不蒸馏 Logit，而是直接用 R1 模型生成的 Chain-of-Thought（CoT）推理数据作为 SFT 训练数据，让小模型（Qwen-7B / LLaMA-8B）学习 R1 的推理过程
  • 本质上是"数据蒸馏"而非传统参数级蒸馏：Teacher 的推理轨迹（长 CoT）比 Teacher 的 Logit 包含了更高层次的"思维过程"信息
  • 结果：DeepSeek-R1-Distill-Qwen-7B 在数学/代码基准上媲美甚至超越直接训练的 GPT-4o mini
💡 区分三种压缩方式：量化（改数值精度，参数数不变）；剪枝（删参数，参数数减少）；蒸馏（小模型模仿大模型输出/行为，训练新模型）。DeepSeek-R1 的"数据蒸馏"是 2025 年最热的新范式，面试必答。

LLM Agent 的 ReAct 框架是如何工作的？Tool Calling（Function Calling）的实现机制是什么？如何解决多步推理中的幻觉问题？
字节 腾讯 中频
▼

📌 参考答案

ReAct（Reason + Act，2022 Yao et al.）：
  • 让 LLM 交替执行"推理（Thought）"和"行动（Action）"步骤，形成 Thought → Action → Observation → Thought... 的循环
  • 与纯 Chain-of-Thought（只推理不行动）相比，ReAct 可以通过真实工具查询获取最新信息（如搜索引擎、代码执行、数据库查询），大幅降低知识截止日期和幻觉问题
Tool Calling（Function Calling）实现：
  • 模型端：在 SFT 阶段训练模型识别何时调用工具，并以特定格式（JSON Schema）输出工具名称和参数；Llama 3 / GPT-4o / Qwen2.5 均原生支持
  • 系统端：将可用工具的 Schema（函数名、参数说明、类型）以 System Prompt 或 Special Token 形式注入；模型输出工具调用 JSON → 应用层拦截 → 执行工具 → 将结果作为 Observation 追加到上下文 → 模型继续推理
  • Parallel Tool Calling：模型一次调用多个工具（如同时查询天气 + 搜索新闻），减少多轮对话的延迟
多步推理中的幻觉问题与缓解：
  • 问题来源：多步推理中前序错误会传播和放大（Error Compounding），且长上下文中模型容易"遗忘"早期约束
  • 自检（Self-Consistency）：采样多条推理路径取多数投票，减少单次采样的随机错误
  • 反思（Reflexion，2023）：在 Agent 失败后，用 LLM 对失败原因进行语言级总结，存储到 Memory 中，下次决策时参考，实现"从错误中学习"
  • 工具验证：对关键事实性输出调用验证工具（代码执行、搜索验证），强制 Grounding
  • Tree-of-Thought / MCTS：将推理树展开，搜索最优路径，减少单线程贪心推理的局部最优陷阱
💡 面试加分：可提及 MCP（Model Context Protocol，Anthropic 提出）作为标准化工具接口协议，以及 A2A（Agent-to-Agent）协议如何让多 Agent 协作，展示对最新生态的了解。

GRPO（Group Relative Policy Optimization）相比 PPO 有什么优势？DeepSeek-R1 如何用强化学习激发大模型的推理能力？
字节 腾讯 必考
▼

📌 参考答案

PPO（Proximal Policy Optimization）在 LLM 对齐中的问题：
  • 标准 RLHF 使用 PPO：需要 Actor（被训练的 LLM）、Critic（估值网络，与 Actor 同等规模）、Reference Model、Reward Model 四个模型同时在显存中，资源消耗巨大
  • Critic 网络训练不稳定，估值偏差会导致策略更新错误
GRPO（DeepSeek-Math 2024 提出）：
  • 去掉 Critic 网络：对同一 Prompt 采样 G 个输出（一组），用这 G 个输出的 Reward 均值作为 Baseline（代替 Critic 的状态值估计），计算相对优势（Relative Advantage）
  • A_i = (r_i - mean(r_1..r_G)) / std(r_1..r_G)——同组内，好的输出获得正优势，差的获得负优势
  • 优势：省去 Critic 模型，显存需求减半；Baseline 估计更稳定（不依赖神经网络拟合）；实现简单
DeepSeek-R1 的 RL 训练策略：
  • Cold Start（冷启动）：先用少量长 CoT SFT 数据微调，防止 RL 初期输出格式混乱
  • GRPO + Verifiable Reward：对数学/代码等可验证问题，直接用答案正确性作为 Reward（不需要 Reward Model），使 Reward 信号精确且无偏
  • Thinking Token：模型在 <think> 标签内自由展开推理过程，RL 鼓励更长更正确的推理链，涌现出自我修正、反思、回溯等行为
  • Rejection Sampling + SFT 迭代：周期性从 RL 模型中采样高质量推理轨迹，加入 SFT 训练集，形成 RL ↔ SFT 交替迭代
核心洞察：纯 RL（不依赖人工标注的 CoT）可以激发大模型的推理涌现能力，Reasoning 能力不是写进参数里的知识，而是可以通过搜索-验证-反馈的强化循环习得的。

MoE（Mixture of Experts）模型的路由机制如何工作？专家负载不均衡（Expert Collapse）如何解决？DeepSeek-V3 的 MoE 设计有何创新？

▼

📌 参考答案

MoE 路由机制：
  • 标准 Sparse MoE 在 FFN 层，每个 Token 由 Router 网络（线性层 + Softmax）选择 Top-K（通常 K=2）个专家，只激活被选中的专家参数进行计算
  • 总参数量 = N 个专家 × 专家大小，但每个 Token 的 FLOPs ≈ K/N × Dense 模型的 FLOPs（激活参数量远小于总参数量）
  • Token Choice（标准做法）：Token 选择 Top-K 专家
  • Expert Choice（2022）：专家选择 Top-K Token（每个专家处理固定数量 Token），天然负载均衡，但 Token 不能保证被处理
专家坍塌（Expert Collapse）问题：
  • 现象：Router 倾向于反复选择少数几个"头部专家"，其余专家几乎不被选中，退化为接近 Dense 模型的效果
  • Auxiliary Loss（辅助负载均衡损失）：在训练损失中加入专家负载均衡正则项，惩罚负载分布不均：L_aux = α × sum(f_i × p_i)，其中 f_i 为专家 i 的 Token 分配比例，p_i 为平均路由概率
  • Expert Dropout：训练时随机丢弃部分专家，迫使 Router 分散依赖
DeepSeek-V3 / V2 MoE 的创新设计：
  • Fine-grained Expert（细粒度专家）：将专家维度缩小但数量增多（如 256 个小专家替代 8 个大专家），Top-K 选 8 个，使路由更细粒度灵活
  • Shared Expert（共享专家）：设置 2 个始终激活的共享专家（处理通用知识），K 个路由专家（处理专业知识），两者输出相加；共享专家减轻了路由专家的负担，降低 Expert Collapse 风险
  • 无辅助损失的负载均衡（DeepSeek-V3）：引入动态 Bias Token 机制（每个专家维护一个可学习的 Bias），通过梯度外的监控自动调整 Bias 平衡负载，避免辅助损失影响主任务性能
  • Multi-Token Prediction（MTP）：每步预测多个 Token（类似 Medusa），额外的 MTP Head 提升训练样本效率，推理时可用于 Speculative Decoding 加速
💡 面试亮点：能说清"共享专家 + 细粒度路由"是 DeepSeek 区别于标准 MoE（GShard/Switch Transformer）的两大工程创新，再提无辅助损失的 Bias 负载均衡机制，展示对最新开源工作的深度理解。

PagedAttention 是如何解决 KV Cache 内存碎片问题的？vLLM 的 Continuous Batching 与传统静态批处理有何本质区别？
字节 腾讯 必考
▼

📌 参考答案

KV Cache 的内存挑战：
  • 自回归解码时，每个 Token 的 Key/Value 需要缓存以避免重复计算，KV Cache 大小与 batch_size × seq_len × num_heads × head_dim × 2 成正比
  • 传统实现为每个请求预先分配 最大序列长度 的连续内存块，即使实际序列很短也占用全量空间 → 内部碎片严重，GPU 利用率低
  • 不同请求的 KV Cache 大小不同，若连续分配会出现 外部碎片，已释放的内存无法被新请求复用
PagedAttention（vLLM 2023，Kwon et al.）：
  • 借鉴操作系统 虚拟内存分页 机制：将 KV Cache 分割为固定大小的 Block（Page）（通常 16 个 Token 的 KV 为一 Block），每个请求的 KV Cache 由若干不连续的 Block 组成
  • 维护 Block Table（类似页表）：记录每个请求的逻辑 Block 号到物理 Block 号的映射，Attention 计算时通过 Block Table 找到实际物理地址
  • 优势：① 内存利用率接近 100%（仅最后一个 Block 有少量内部碎片）；② 支持 Prefix Sharing（多个请求共享相同前缀的 Block，写时复制 CoW）；③ 支持 Beam Search 的 Block 分叉
Continuous Batching（迭代级调度）vs 静态批处理：
  • 静态批处理：一批请求等所有请求都生成完毕才算一批完成，最短的请求生成完后仍需等待最长请求 → GPU 存在大量空泡（Bubble），短请求浪费等待时间
  • Continuous Batching：每生成一个 Token 后（迭代级），调度器检查哪些请求已完成（遇到 <EOS>），立即从等待队列中插入新请求填补空位，始终维持 GPU 高满负荷状态
  • 效果：吞吐量提升 2-4× （相比 Orca 论文中的静态批处理基线），响应延迟更低，尤其对长短混合请求效果显著
vLLM 推理服务的整体架构：LLM Engine → Scheduler（连续批处理调度）→ Worker（PagedAttention GPU 计算）→ Block Manager（KV Cache 分页管理）。2024 年新增 Chunked Prefill（Prefill 分块执行，减少首 Token 延迟对 Decode 请求的干扰）。
💡 类比记忆：PagedAttention = KV Cache 的"虚拟内存"；Continuous Batching = LLM 推理的"流水线调度"。两者组合让 GPU 吞吐率不再受限于最长序列的"木桶效应"。

EAGLE 投机解码的原理是什么？与标准 Draft-Then-Verify 投机解码相比，EAGLE 的关键改进是什么？
字节 腾讯 中频
▼

📌 参考答案

标准投机解码（Speculative Decoding，Leviathan 2022）回顾：
  • 用小 Draft Model 快速生成 K 个候选 Token，再用大 Target Model 并行验证；验证通过的 Token 无需逐一生成，等效于一次 Forward 得到多个 Token
  • 加速比 = 平均接受 Token 数 / 单次 Target Forward 耗时；加速瓶颈在于 Draft Model 的接受率
  • 问题：Draft Model 是独立小模型，与 Target Model 分布存在偏差，接受率（Acceptance Rate）受限（通常 70-85%）
EAGLE（2024，SafeAI Lab）的关键洞察：
  • 不使用独立小模型作为 Draft，而是利用 Target Model 自身 倒数第二层的特征向量（Feature）来预测下一个 Token 的特征，从而构建 Draft
  • Draft 网络极轻量（仅一个 Transformer Block + LM Head），以 Target Model 的 Feature 为输入，自回归地预测未来多步特征 → 再解码为 Token
  • 由于 Draft 直接在 Target Model 的特征空间操作，接受率显著提升（>90%），加速比 2.5-3.5× （相比 Vanilla 解码）
EAGLE-2（2024）的进一步改进：
  • 引入 动态草稿树（Dynamic Draft Tree）：不再使用固定深度 K 的线性候选序列，而是根据 Draft Token 的置信度（Confidence）动态扩展树结构；置信度高的路径深度展开，置信度低的路径剪枝
  • Context-Aware Dynamic Draft：在输入上下文变化时自适应调整树的分支结构，进一步提升接受率（比 EAGLE-1 再提升约 20%）
Medusa vs EAGLE：
  • Medusa：在 LM Head 旁并联多个"草稿头"（Medusa Heads），每个头预测不同位置的 Token；无自回归依赖，速度更快但接受率略低
  • EAGLE：自回归草稿 + Feature-level 对齐，接受率更高，但有一个 Block 的串行草稿延迟
💡 面试要点：EAGLE 的核心是"在 Target Model 自己的特征空间里做草稿预测"，而非用外部小模型，这是接受率高的根本原因。能说清 Feature-Level Draft + Tree Attention 验证两步，就掌握了 EAGLE 的本质。

RAG 进阶：HyDE、Self-RAG 和 Corrective RAG（CRAG）各解决了什么问题？Agentic RAG 与传统 Naive RAG 的核心区别是什么？
字节 腾讯 必考
▼

📌 参考答案

Naive RAG 的痛点：用户 Query 直接做向量检索，但 Query 往往简短或措辞与文档不匹配，导致召回质量差（语义鸿沟），生成内容可能与检索结果脱节或检索到无关文档。

HyDE（Hypothetical Document Embeddings，2022）：
  • 先让 LLM 根据 Query 生成一篇假设性答案文档（Hypothetical Document），再用该文档的 Embedding 去检索知识库，而非用原始 Query Embedding
  • 原理：假设文档与真实文档在 Embedding 空间更接近（两者都是"答案形态"），比 Query（"问题形态"）检索更准确
  • 缺点：生成假设文档有额外 LLM 调用延迟；若 LLM 幻觉严重，假设文档偏差导致检索偏移
Self-RAG（2023，Asai et al.）：
  • 训练 LLM 主动决策：① 是否需要检索（而非每次都检索）；② 检索到的文档是否相关（Relevance 判断）；③ 生成内容是否有文档支撑（Support 判断）；④ 最终输出是否有用（Utility 判断）
  • 引入特殊 Reflection Token（[Retrieve] / [IsREL] / [IsSUP] / [IsUSE]），以 Token 形式嵌入生成流程，实现端到端的自我批判
  • 优势：按需检索节省延迟；能拒绝不相关文档，输出更可靠
Corrective RAG（CRAG，2024）：
  • 引入 检索质量评估器：对检索到的文档打分（CORRECT / AMBIGUOUS / INCORRECT）
  • 若文档质量不足，自动触发 Web Search 补充检索（外部知识扩充），再对检索内容做 Knowledge Refinement（提取关键段落，过滤噪声）
  • 与 Self-RAG 互补：Self-RAG 在生成侧批判，CRAG 在检索侧纠错
Agentic RAG 与 Naive RAG 的核心区别：
  • Naive RAG：Query → 单次检索 → 单次生成，线性固定流程
  • Agentic RAG：LLM 作为 Agent 自主规划检索策略，可以多步检索（根据中间结果决定下一步查什么）、调用多个知识源（向量库 + Web + 代码执行）、对检索结果进行推理合成，形成循环迭代直到满足停止条件
  • 典型模式：Plan → Retrieve → Reason → Re-Retrieve → Generate，代表工具：LangGraph、LlamaIndex Workflow
💡 面试框架：RAG 演进 = 解决"检索质量"问题（HyDE/CRAG）+ 解决"何时检索"问题（Self-RAG）+ 解决"复杂多步推理"问题（Agentic RAG）。三个维度各有代表方案，清晰对应考点。

LLM 大规模分布式训练中，ZeRO 优化、Tensor 并行和流水线并行分别解决什么问题？3D 并行策略如何组合使用？
字节 腾讯 必考
▼

📌 参考答案

大模型训练的内存瓶颈分析：
  • 以 70B 模型 BF16 训练为例：参数 ~140GB，Adam 优化器状态（m/v/master weight）~420GB，激活值（根据 batch size 可达数百 GB），总需求远超单卡显存（80GB A100）
ZeRO（Zero Redundancy Optimizer，DeepSpeed）：
  • 核心思想：消除数据并行（DP）中每张卡存储相同副本的冗余
  • ZeRO Stage 1：分片优化器状态（Optimizer States Partition）——每卡只存 1/N 的 Adam m/v，反向传播后 AllGather 汇总
  • ZeRO Stage 2：+ 分片梯度（Gradient Partition）——减少梯度 AllReduce 通信量
  • ZeRO Stage 3：+ 分片参数（Parameter Partition）——参数也分片存储，前向/反向时按需 AllGather，内存最省但通信开销最大
  • ZeRO-Offload：将优化器状态 Offload 到 CPU 内存，GPU 只保留参数和激活
Tensor 并行（Megatron-LM）：
  • 将单层的权重矩阵 按列/行切分 到多卡（如 8 卡 TP），每卡计算部分 Attention Head 或 FFN 维度，再 AllReduce 汇聚
  • 优势：减少单层内存；劣势：需要低延迟高带宽互联（NVLink 内部），跨机器 TP 效率差
  • 通常限制在 单节点内（8 卡）做 TP
流水线并行（Pipeline Parallel）：
  • 将模型按 层（Layer） 划分到不同节点，数据像流水线一样流过各阶段
  • GPipe（Micro Batch）：将 Batch 切成 Micro Batch，填充流水线气泡，但需缓存所有 Micro Batch 的激活
  • 1F1B（One Forward One Backward，Megatron PP）：交替执行 Forward/Backward，激活内存减少，Pipeline Bubble 约 1/(m+1)（m 为 Micro Batch 数）
3D 并行策略：
  • TP × PP × DP 三维组合：TP 在机内节点内（NVLink），PP 跨节点层划分，DP 跨节点数据复制（ZeRO 优化显存）
  • 典型配置（1024 卡训练 70B）：TP=8（单机内），PP=8（8 台机器），DP=16（共 8×8×16=1024 卡）
  • Sequence Parallel（SP）：在 TP 的基础上，将 LayerNorm/Dropout 沿序列维度分片，进一步减少激活内存
💡 记忆口诀：ZeRO 解决"数据并行冗余显存"；Tensor 并行解决"单层权重太大"；Pipeline 并行解决"层数太多跨节点"。3D 并行 = 三者叠加，是训练百亿参数模型的工业标配。

FLUX 图像生成模型的架构有何创新？Flow Matching 与 DDPM/DDIM 相比有什么优势？
字节 网易 中频
▼

📌 参考答案

FLUX（Black Forest Labs，2024）架构创新：
  • 双流 Transformer（Double Stream）：图像 Token 和文本 Token 在 Transformer 块中各维护独立的流（Stream），通过交叉注意力交互，而非将两者拼接到同一序列；更好地保留各模态的独立特征表示
  • MM-DiT（Multi-Modal DiT）：基于 Stable Diffusion 3 的 MM-DiT 改进，每个块中图像和文本互相 Cross-Attend，然后各自经过 FFN 更新
  • RoPE 位置编码用于图像 Patch：图像 Patch Token 使用二维 RoPE 编码位置，支持任意分辨率和宽高比的生成
  • Flow Matching 训练目标（区别于传统 DDPM 噪声预测）
Flow Matching（Lipman et al. 2022 / Albergo & Vanden-Eijnden）：
  • 传统 DDPM：前向过程逐步加高斯噪声（马尔可夫链），后向过程预测噪声 ε；路径是弯曲的随机游走
  • Flow Matching：定义从噪声 x_1 到数据 x_0 的 直线插值路径（Rectified Flow / Linear ODE）：x_t = t·x_0 + (1-t)·x_1，训练模型预测速度场 v_t
  • 优势：
    • 路径更短更直，采样步数更少（8-20 步即可，DDPM 需 1000 步）
    • ODE 求解器可以用更大步长（Euler / Heun），减少 NFE（Number of Function Evaluations）
    • 训练更稳定（不依赖噪声调度表 β_t 的精细设计）
    • 理论更简洁：直接优化 CFM（Conditional Flow Matching）的 MSE 损失
SDXL vs FLUX 对比：
  • SDXL：U-Net + DDPM/DDIM，仍是卷积主导的架构；在文本细节（手、文字）上较弱
  • FLUX.1-dev（12B 参数）：全 Transformer DiT + Flow Matching；文字生成、手部细节、提示词遵循度均优于 SDXL；同等步数质量更高，已成为 2024-2025 年最主流的开源图像生成模型
💡 一句话总结：FLUX = MM-DiT 双流架构（更好图文交互）+ Flow Matching 训练（更直的路径、更少采样步）+ RoPE 位置编码（任意分辨率）。三个关键词记住就能说清楚 FLUX 相比 SD 系列的核心进步。

Transformer 的 Pre-LN 与 Post-LN 有什么区别？为什么现代大模型几乎都用 Pre-LN？RMSNorm 相比 LayerNorm 有何优势？
腾讯 字节 必考
▼

📌 参考答案

Post-LN（原始 Transformer，Vaswani 2017）：
  • 结构：x = LayerNorm(x + Sublayer(x))，归一化放在残差连接之后
  • 问题：残差通路在归一化之前，梯度需要穿过 LayerNorm 才能到达主流，导致深层 Transformer（>12 层）训练时 梯度消失/爆炸 严重，需要 Warmup + 精细调参才能收敛
Pre-LN（GPT-2 / LLaMA 等现代大模型）：
  • 结构：x = x + Sublayer(LayerNorm(x))，归一化放在 Sublayer 之前，残差路径直通（无 LN 阻隔）
  • 优势：残差路径梯度流畅，训练更稳定，不需要精细 Warmup，支持更深的网络；但理论上最后一层输出未经 LN 归一化，表示空间有轻微漂移（通常最后加一个 Final LN 解决）
  • 几乎所有 2020 年后的大模型均采用 Pre-LN 或其变体（DeepNorm、RMSNorm Pre-LN）
RMSNorm（Root Mean Square Normalization，Zhang & Sennrich 2019）：
  • LayerNorm 公式：y = (x - μ) / (σ + ε) × γ + β，需要计算均值 μ 和方差 σ，以及可学习的 β 偏置
  • RMSNorm 简化：去掉均值归一化和偏置 β，只保留 RMS 缩放：y = x / RMS(x) × γ，其中 RMS(x) = sqrt(mean(x²) + ε)
  • 优势：
    • 计算量减少约 7-15%（不需要计算均值，不需要减均值）
    • 在大规模训练中效果与 LayerNorm 相当，实际质量损失可忽略
    • 更简洁的实现，FP16/BF16 下数值更稳定
  • 采用方：LLaMA 全系列、Qwen 全系列、Mistral、DeepSeek 全系列均使用 RMSNorm Pre-LN
DeepNorm（微软，Transformer 深度突破）：对 Post-LN 加入额外的 α 缩放残差（x = LayerNorm(α·x + Sublayer(x))），配合特殊初始化，使 1000 层 Post-LN Transformer 可以稳定训练，但工程上复杂度更高，实际应用不及 RMSNorm Pre-LN 广泛。
💡 面试高频考点："为什么 LLaMA 用 RMSNorm 不用 LayerNorm？"——答：① 计算量更小（省去均值计算和偏置）；② 效果相当；③ 更简洁。同时说 Pre-LN 的训练稳定性优势，两点串联即是完整答案。

SGLang 的 RadixAttention 如何实现细粒度 Prefix KV Cache 共享？与 vLLM 的 Prefix Caching 相比有何优势？Cache-Aware Scheduling 的作用是什么？

Prefix Caching 的动机：
  • 多轮对话、RAG 场景中，大量请求共享相同的前缀（System Prompt、文档上下文），这部分 Token 的 KV Cache 可以复用，避免重复 Prefill 计算
  • vLLM 的 Prefix Caching：以固定 Block（16 Token）为单位做 Hash，前缀 Hash 匹配则复用该 Block 的 KV Cache；粒度较粗，只有完整 Block 对齐的前缀才能命中
RadixAttention（SGLang 核心创新）：
  • 使用 Radix Tree（基数树）存储所有请求的 KV Cache，树的每条边对应一段 Token 序列，节点保存该前缀对应的物理 KV Cache 块
  • 细粒度前缀匹配：沿树路径做最长公共前缀（LCP）搜索，任意长度的公共前缀都可命中，不受 Block 对齐限制（相比 vLLM 的 Block Hash，匹配粒度更精细）
  • 写时复制（CoW）：多请求共享同一前缀节点时，节点引用计数递增；某请求需要修改时才复制，避免写冲突
  • LRU 驱逐策略：树节点按最近使用时间排序，内存不足时驱逐最久未用的叶节点（类似 CPU Cache 的 LRU）
Cache-Aware Scheduling：
  • 调度器在分配新请求时，优先选择与当前 Radix Tree 中命中率最高的请求优先处理（Hit-Rate-First Scheduling），最大化 KV Cache 命中，减少重新 Prefill 的开销
  • 对多租户（Multi-Tenant）推理尤其有效：不同用户的 System Prompt 相同时，命中率接近 100%，整体 TTFT（Time To First Token）大幅降低
SGLang vs vLLM 综合对比：
  • RadixAttention 在 Multi-Turn / RAG / Shared Prefix 场景吞吐量比 vLLM 高 1.5-5×；单请求短文本场景两者接近
  • SGLang 还引入 Constrained Decoding（基于有限状态机约束 JSON/Regex 输出），减少结构化输出的无效 Token 生成
💡 类比记忆：RadixAttention = KV Cache 的"文件系统目录树"，公共路径自动共享，节点引用计数管理生命周期；Cache-Aware Scheduling = 让调度器"优先处理命中缓存的请求"，类似 CPU Cache 的局部性原理。

Mamba / SSM（状态空间模型）的核心原理是什么？与 Transformer 自注意力相比有何优劣？Sliding Window Attention 和 Sparse Attention 又分别如何缓解长序列的 O(N²) 问题？
字节 腾讯 必考
▼

📌 参考答案

Transformer 自注意力的复杂度问题：
  • 自注意力计算 QK^T 的复杂度为 O(N²)，N 为序列长度；1M Token 上下文的注意力矩阵需要 ~2PB 内存，完全不可行
SSM（State Space Model）/ Mamba（Gu & Dao，2023）：
  • 核心思想：用连续时间状态空间方程 h'(t) = Ah(t) + Bx(t)，y(t) = Ch(t) 描述序列依赖，将其离散化为循环形式（类似 RNN），单步推理 O(1) 状态大小
  • 选择性机制（S6）：Mamba 引入输入相关的 A/B/C 矩阵（动态权重），使 SSM 具备与内容相关的选择能力（普通 SSM 的 A/B/C 是固定的），弥补了传统 SSM 无法"忘记"无关信息的缺陷
  • 硬件感知并行训练：推理时串行（O(N) 时间，O(1) 状态），训练时可通过并行扫描（Parallel Scan）算法并行化，配合 FlashAttention 类似的 IO-Aware CUDA Kernel 实现高效训练
  • 优势：① 推理内存恒定（无 KV Cache 无限增长问题）；② 长序列吞吐量高（O(N) vs Transformer O(N²)）
  • 劣势：① 长距离精确信息检索（In-Context Learning）能力弱于 Transformer；② 对"任意位置 Attention"的需求场景（如 RAG 精确匹配）效果略差
Sliding Window Attention（SWA）：
  • 每个 Token 只与其前后 W 个 Token 做 Attention（W 为窗口大小），复杂度降至 O(N×W)
  • 通过多层堆叠，底层局部窗口的信息可以逐层传播，感受野以 L×W 扩展（L 为层数）；Mistral-7B 使用 SWA，局部窗口 4096，有效上下文通过层传播可达 32K+
  • 缺点：无法直接 Attend 到远处非窗口内的 Token，对需要全局信息的任务有损失
Sparse Attention（BigBird / Longformer）：
  • 结合三类注意力模式：① 局部窗口注意力（Local Window）；② 全局 Token 注意力（CLS 等特殊 Token Attend 所有位置）；③ 随机稀疏注意力（Random Attention）
  • 理论上任意两 Token 间信息都可通过 O(log N) 跳全局 Token 传递，同时保持 O(N) 复杂度
混合架构趋势：2024-2025 年主流方案是 Hybrid Mamba-Transformer（如 Jamba、Zamba），在 Full Attention 层之间插入 Mamba 层，长序列用 SSM 处理，关键跨位置依赖由少量 Attention 层处理，兼顾效率与质量。
💡 核心对比：Transformer（全局 O(N²)，精确，KV Cache 随序列增长）→ SWA（局部 O(NW)，有损）→ Mamba（递归 O(N)，无 KV Cache，长距检索弱）→ Hybrid（折衷最优）。这条演进线串起了长序列建模的主流技术路线。

▼

📌 参考答案

早期 VLM 的固定分辨率局限（LLaVA-1.5 等）：
  • 将图像 Resize 到固定分辨率（如 336×336），经 ViT 编码后得到固定数量的视觉 Token（如 256 个），通过 MLP 投影层连接至 LLM
  • 高分辨率图像（文档、图表）强行压缩后细节丢失严重，OCR 任务效果差
动态分辨率方案：
  • InternVL 系列（Dynamic High Resolution）：将高分辨率图像切分为多个固定大小（448×448）的 Tile，每个 Tile 独立经 ViT 编码；额外生成一个 Thumbnail（缩略图）提供全局上下文；所有 Tile 的 Token 拼接后送入 LLM。Tile 数量动态（最多 12 个），总视觉 Token 数随图像复杂度变化
  • Qwen2.5-VL（Naive Dynamic Resolution）：直接将不同分辨率图像的 Patch 映射为不同数量的视觉 Token，无需固定 Tile；引入 2D-RoPE（MRoPE） 编码每个 Patch 的二维位置信息，LLM 侧直接 Attend 变长视觉 Token 序列；视频帧同样支持动态 Token 数（按帧率/时长采样）
ViT-LLM 连接层设计对比：
  • MLP Projector（LLaVA）：两层线性变换，简单高效；视觉 Token 逐一映射到 LLM 维度，Token 数不变
  • Pixel Shuffle / C-Abstractor（InternVL）：将 2×2 相邻视觉 Token 合并为 1 个（空间降采样 4×），同时通道数扩大，减少 LLM 侧处理的 Token 数（从 1024 → 256），降低计算量，效果优于简单 Pooling
  • 原生集成（Qwen2.5-VL）：ViT 与 LLM 共同训练，视觉 Token 直接在 LLM Attention 中与文本 Token 交互，无独立投影层；MRoPE 使视觉 Token 的位置信息自然融入
视觉 Token 压缩的必要性：
  • 高分辨率图像经 ViT 编码后视觉 Token 可达 1000-4000 个，与文本 Token 拼接后序列长度极大，Attention 计算量 O(N²) 急剧增加
  • 压缩方案：Spatial Pooling（平均池化）、Token Merging（相似 Token 合并）、Perceiver Resampler（固定数量 Query 蒸馏视觉信息）；InternVL 的 Pixel Shuffle 是工程上最常用的高效方案
💡 2025 VLM 发展趋势：动态分辨率 + 原生多模态训练（ViT 与 LLM 联合）+ 视频理解（时序 Token 管理）是主流方向；InternVL 和 Qwen-VL 系列是国内最有影响力的两条开源路线，面试必须能说清两者架构异同。

ControlNet 的 Zero Convolution 机制为何能在训练初期不破坏基础扩散模型？IP-Adapter 与 ControlNet 的条件注入方式有何本质区别？
字节 网易 中频
▼

📌 参考答案

ControlNet 核心设计（Zhang et al. 2023）：
  • 在 Stable Diffusion 的 U-Net Encoder 旁添加一个可训练副本（Trainable Copy），接受控制条件（如 Canny 边缘图、Depth Map、Pose 关键点）作为额外输入
  • 副本的输出通过加法注入到 SD U-Net 的对应 Decoder 层
Zero Convolution（零卷积）的作用：
  • 连接控制分支与主干 U-Net 的卷积层，其权重初始化为全零，偏置也初始化为零
  • 训练初期效果：权重全零 → Zero Conv 输出全零 → 副本注入主干的贡献为零 → 主干 U-Net 完全不受影响，仍能正常生成图像（生成能力不损坏）
  • 随着训练进行，Zero Conv 权重逐渐从零开始学习，控制分支的影响慢慢注入；整个过程是"从无到有"的渐进式影响，避免了训练初期梯度爆炸或模型崩溃
  • 相比之下，若直接用普通随机初始化的卷积连接，训练初期随机噪声注入主干，会立刻破坏预训练权重
ControlNet 支持的条件类型：Canny/HED 边缘图、Depth/Normal Map、OpenPose 人体姿态、Segmentation Map、Scribble、Line Art 等，每种条件需单独训练对应 ControlNet 权重。

IP-Adapter（腾讯 ARC Lab，2023）：
  • 目标：以参考图像风格/内容为条件控制生成，而非几何结构（区别于 ControlNet）
  • 核心：引入解耦的 Cross-Attention 模块，文本特征走原有的 Cross-Attention，图像特征（CLIP Image Encoder 提取）走独立的新增 Cross-Attention，两者各自的 K/V 不互相干扰
  • 仅训练新增的 Cross-Attention 层（约 22M 参数），SD U-Net 主干权重冻结，参数高效
ControlNet vs IP-Adapter 本质区别：
  • ControlNet：控制空间结构/几何（边缘、深度、姿态），通过残差加法注入 Decoder 层；控制图与生成图空间位置严格对应
  • IP-Adapter：控制外观风格/语义内容（参考图的风格、人脸、物体），通过独立 Cross-Attention 注入；不严格绑定空间位置，侧重"看起来像参考图"
  • 两者可组合使用：ControlNet 控制姿态 + IP-Adapter 控制角色外观，实现精确控制生成
💡 一句话区分：ControlNet = "照着这张结构图生成"（几何约束）；IP-Adapter = "生成一张风格像这张图的图"（外观约束）。Zero Conv 的作用是"从零开始学影响，训练初期不干扰主干"。

llama.cpp / GGUF 的量化部署与 GPTQ/AWQ 的 GPU 量化有何区别？INT4 量化为什么能大幅降低显存而精度损失较小？量化感知训练（QAT）与训练后量化（PTQ）的适用场景？
腾讯 字节 必考
▼

📌 参考答案

量化的基本原理：
  • 将浮点权重/激活值映射到低位整数（INT8/INT4/INT2），减少内存和带宽需求
  • 对称量化：q = round(x / scale)，scale = max(|x|) / 127；非对称量化：q = round(x / scale) + zero_point，可处理非对称分布
  • INT4 理论上比 FP16 节省 4× 显存，带宽需求降至 1/4，适合 Memory-Bound 的自回归解码阶段
GPTQ（GPU PTQ，2022）：
  • Layer-wise 训练后量化，使用 OBQ（Optimal Brain Quantization）算法：按列顺序量化权重，每量化一列立刻更新其余列的权重来补偿误差（基于 Hessian 二阶信息）
  • 量化为 INT4 时 Perplexity 损失约 0.1-0.5，实用。仅需少量校准数据（128 序列），速度较快
AWQ（Activation-Aware Weight Quantization，2023）：
  • 观察到权重中约 1% 的"显著通道"（Salient Channels）对精度影响极大，这些通道对应激活值较大的输入通道
  • 对显著通道使用较大的缩放因子（Scale-Up）保护精度，其他通道正常量化；通过搜索最优 Scale 因子实现，无需反向传播
  • 比 GPTQ 精度更高（尤其 INT4），且支持更高效的 Kernel（TinyChat / AWQ CUDA Kernel）
GGUF / llama.cpp 量化：
  • 面向 CPU/低端 GPU 的边端部署，支持 Q2_K/Q4_K/Q5_K/Q8_0 等多种量化精度
  • 采用分组量化（Group Quantization）：每 32/64 个权重共享一组 Scale/Zero，减少量化误差；GGUF 格式将模型结构和量化参数打包为单文件，便于分发
  • 支持混合精度（重要层用 Q8，其他层用 Q4），平衡速度和精度
  • 主要差异：GPTQ/AWQ 面向 CUDA GPU（A100/H100）高性能服务；GGUF 面向边端 CPU 甚至 Apple M 系列 Metal GPU
QAT（量化感知训练）vs PTQ（训练后量化）：
  • PTQ（GPTQ/AWQ/GGUF 均属此类）：在已训练好的模型上直接量化，速度快，无需重训练；适合大模型（70B+）快速量化部署，精度损失 INT8 可接受，INT4 有轻微下降
  • QAT：在训练时引入模拟量化节点（Fake Quantize），反向传播时使用 STE（直通估计器）绕过不可微的取整操作；模型主动学习在量化约束下工作，精度最好，但需要完整训练流程，计算成本高；适合小模型（1-7B）且对精度要求极高的场景（端侧手机部署 INT4）
💡 工程选型口诀：服务端大模型 → AWQ/GPTQ（GPU INT4）；端侧手机 CPU → GGUF Q4_K_M；对精度极敏感的小模型 → QAT INT8。说清"PTQ 速度快但精度略损，QAT 精度好但要重训"，就覆盖了面试核心考点。

LLM Function Calling / Tool Use 的完整实现机制是什么？结构化输出（JSON Schema 约束）如何通过有限状态机（FSM）保证格式正确性？与 ReAct Agent 的关系？
字节 腾讯 必考
▼

📌 参考答案

Function Calling 的完整流程：
  • ① 工具声明：调用方将可用工具的 JSON Schema（函数名、参数名、类型、描述）注入 System Prompt 或特殊 Tool Definition 格式
  • ② 模型决策：LLM 根据用户请求判断是否需要调用工具，若需要则输出结构化 Tool Call JSON（{"name": "get_weather", "arguments": {"city": "北京"}}），而非直接回答
  • ③ 工具执行：调用方解析 Tool Call JSON，实际执行对应函数（API 调用、数据库查询等），获取结果
  • ④ 结果回传：将工具执行结果以 Tool Result 角色插入 Conversation History，LLM 继续生成最终回答
  • ⑤ 多轮工具调用：复杂任务中 LLM 可连续调用多个工具，每次调用后根据结果决定下一步
训练侧支持：专门训练的 Function Calling 模型（如 GPT-4o、Qwen2.5、Llama 3.1）在 SFT 阶段使用大量工具调用示例，学习"何时该用工具"和"如何生成合法 JSON"。

结构化输出 / JSON Schema 约束（Constrained Decoding）：
  • 问题：LLM 生成的 JSON 可能格式错误（漏括号、类型不匹配），导致解析失败
  • 有限状态机（FSM）约束解码：将 JSON Schema 编译为 FSM，每个状态表示当前合法的下一个字符集合；每个解码步骤只允许在当前状态合法的 Token 中采样（对非法 Token 的 Logit 设为 -inf），保证输出 100% 符合 Schema
  • 代表实现：Outlines（Python 库）/ SGLang Constrained Decoding / llama.cpp Grammar Sampling（使用 GBNF 文法）
  • 代价：编译 FSM 有一次性开销（通常 <1ms），解码时每步 Token Mask 计算 ~0.1ms；对性能影响极小
Function Calling vs ReAct Agent：
  • Function Calling：侧重单步工具调用的精确格式保证，是技术实现层面的机制
  • ReAct（Reason + Act，Yao et al. 2022）：是一个 Agent 框架范式，LLM 交替产生 Thought（推理）和 Action（工具调用），观察工具结果后继续 Thought，形成循环直到任务完成
  • 关系：ReAct 框架的 Action 步骤通常通过 Function Calling 机制实现；Function Calling 提供调用的格式保障，ReAct 提供多步推理的流程框架
💡 面试核心：Function Calling = 让 LLM 输出结构化 Tool 调用 + 外部系统执行 + 结果回传；Constrained Decoding = 用 FSM 保证 JSON 格式正确，对非法 Token 屏蔽 Logit；ReAct = 多步 Thought-Action-Observation 循环，Function Calling 是 ReAct Action 的底层实现。

DiT（Scalable Diffusion Models with Transformers，Peebles 2023）核心思想：用 Transformer 替换扩散模型中的 UNet backbone，在 Latent 空间（LDM 风格）做去噪，验证了扩散模型的 Scaling Law。

关键架构设计：
  • Patch Embedding：将 Latent（如 32×32）按 p×p 划分 Patch（默认 p=2 或 4），线性展平后加 2D sin-cos 位置编码，转为 token 序列送入 Transformer
  • adaLN-Zero（Adaptive LayerNorm）：条件信号（Timestep t + Class Label c）先经 MLP 生成 γ、β、α 六个调制参数；LayerNorm 后用 γ/β 做仿射变换，残差分支乘 α（初始化为 0，使初始网络等效恒等变换）
  • Timestep & Class 条件注入：t 做 Sinusoidal Embedding → MLP；c 做 Class Embedding → MLP；两者相加后统一调制每个 DiT Block，比 Cross-Attention 注入更轻量
  • Final Layer：用 adaLN 处理最后一层后，Linear Decode 回 (p×p×2C) 维，分为 ε（噪声预测）和 Σ（方差预测）两部分
DiT vs UNet（Score 模型 backbone）：
  • UNet：Skip Connection 携带多尺度局部特征，适合像素级细节；参数量有上限，Scaling 困难
  • DiT：全 Transformer，全局 Self-Attention 自然建模长距离依赖；参数量随深度/宽度线性可扩；FLOPs 增大时 FID 持续下降，验证了 Scaling 规律
  • DiT-XL/2（最大变体）在 ImageNet 256×256 FID=2.27，首次超越 ADM-UNet
后续演进：Sora（OpenAI）将 DiT 扩展到视频（Space-Time Patch）；FLUX 使用双流 MM-DiT（文本与图像各自 Self-Attention + Cross-Attention 融合）；Stable Diffusion 3 也采用 MM-DiT 架构。
💡 一句话：DiT = LDM 的 UNet 换成 Transformer + adaLN-Zero 条件注入 + Patch Embedding；核心贡献是证明扩散模型的 Gflops-FID Scaling Law。

GAN vs Diffusion Model：生成机制、训练稳定性、多样性/质量对比及各自适用场景
腾讯 字节 中频
▼

📌 参考答案

生成机制对比：
  • GAN（生成对抗网络）：Generator 从噪声直接生成样本，Discriminator 判别真假；二者博弈驱动 G 学习数据分布；单步推理（Generator forward 一次即得结果），推理速度极快
  • Diffusion Model：Forward 过程逐步向数据加高斯噪声（T 步），Reverse 过程用神经网络逐步去噪；推理需要 20~1000 步 Denoising（DDIM/DPM-Solver 可缩短到 10~50 步），速度较慢
训练稳定性：
  • GAN：博弈导致 Mode Collapse（生成单一模式）、梯度消失（Discriminator 过强时）、训练不收敛等问题；需要仔细调超参、学习率、Spectral Norm 等技巧
  • Diffusion：只需训练一个去噪网络，目标是简单的 MSE 噪声预测；无对抗博弈，训练极稳定，但收敛慢（大量扩散步骤）
生成质量 & 多样性：
  • GAN：FID 曾是 SOTA，但多样性有限（Mode Coverage 不足）；Precision 高但 Recall（多样性覆盖）低
  • Diffusion：多样性显著优于 GAN（完整覆盖数据分布），样本质量在大规模预训练后全面超越 GAN；2022 年后成为 SOTA
适用场景：
  • GAN 仍占优：实时生成（游戏 Avatar、人脸交换、实时风格迁移，需单步推理）、Super Resolution（ESRGAN）、医学图像（数据增强）
  • Diffusion 占优：高质量图文生成（SDXL/FLUX/Midjourney）、视频生成（Sora）、3D/4D 生成、科学分子设计（FoldDiff）等质量优先场景
混合趋势：Consistency Model（单步扩散蒸馏，速度接近 GAN）、LCM、Lightning 等让 Diffusion 在速度上逼近 GAN，进一步缩小差距。
💡 记忆对比：GAN = 快但不稳，Diffusion = 稳且强但慢。现代 Diffusion 蒸馏（Consistency / Rectified Flow）已大幅压缩推理步数，GAN 在大规模图像生成领域的优势基本消失。

LLM KV Cache 管理进阶：驱逐策略（H2O / StreamingLLM / SnapKV）与长上下文优化
字节 腾讯 高频
▼

📌 参考答案

KV Cache 为什么需要驱逐：Transformer Decoder 推理时 KV Cache 随上下文线性增长，100K token 的 70B 模型 KV Cache 可达数十 GB，超出单卡显存；需要在不重新计算的前提下丢弃不重要的 token KV。

主流驱逐策略：
  • H2O（Heavy Hitter Oracle）：统计注意力分数累积 Hit 次数，保留累积 Attention Score 最高的 "Heavy Hitter" Token；原理：少数 Token 承担了大部分注意力（Power Law），保留它们可最小化信息损失；代价：需要动态统计，硬件并行度下降
  • StreamingLLM（Attention Sink）：发现 Attention 分数中初始 Token（Sink Token，通常是 BOS）始终获得极高分数（即使语义无关），原因是 Softmax 归一化强制某些 Token 成为"概率收纳器"；解决方案：始终保留 Sink Token（4 个）+ 滑动窗口内近期 Token，无限流式推理而不退化
  • SnapKV：在 Prefill 阶段对每个 Head 统计 Query 的注意力分布热点，提前确定哪些 KV 位置重要，在 Prefill 末端一次性压缩 KV Cache，减少 Decode 阶段内存占用；精度优于简单 Sliding Window
与 PagedAttention（vLLM）的区别：PagedAttention 解决的是 KV Cache 的碎片化分配（物理内存管理），驱逐策略解决的是超长序列超出显存的问题；二者可以组合使用。

长上下文专项优化：
  • MLA（DeepSeek-V2/V3）：低秩压缩 KV 维度，KV Cache 降低 ~5×，同时保留性能
  • GQA/MQA：多 Query 共享少量 KV Head，Cache 大小减少 Group Size 倍
  • 量化 KV Cache（KVQuant / KVSharer）：将 KV 量化为 INT4/INT8，显存减半但精度损失可控
💡 面试要点：驱逐策略核心问题 = "丢哪些 Token 损失最小"。H2O 保留累积高注意力；StreamingLLM 保留 Sink Token + 近期窗口；SnapKV 在 Prefill 提前压缩。三者都是 Training-Free，可直接部署在现有模型。

LLM 推理框架全景：vLLM vs SGLang vs TensorRT-LLM vs Triton Inference Server — 架构差异与选型
字节 腾讯 高频
▼

📌 参考答案

四大框架定位：
  • vLLM（UC Berkeley）：业界标准开源框架；核心创新 PagedAttention（分页 KV Cache）+ Continuous Batching；支持最广泛的模型格式；适合：开源模型通用部署、研究原型、中等吞吐量场景
  • SGLang（UC Berkeley）：面向复杂 LLM 程序（多轮对话、Tree-of-Thought、RAG）；核心创新 RadixAttention（KV Cache Prefix 树复用）+ 编译器优化（SGLang DSL）；Prefix 缓存命中率高时吞吐量显著超越 vLLM；适合：Prefix 长且共享度高的 Serving（Few-Shot / System Prompt 固定场景）
  • TensorRT-LLM（NVIDIA）：极致性能优化；将模型编译为 TensorRT Engine，融合算子（Fused QKV / Fused MLP）、INT8/FP8 量化、In-flight Batching；硬件强绑定（只支持 NVIDIA GPU）；适合：NVIDIA 生产环境极致吞吐量（P/T100 系列）
  • Triton Inference Server（NVIDIA）：通用推理服务框架，支持多后端（TensorRT / ONNX / PyTorch / OpenVINO）；提供 gRPC/HTTP API、动态批处理、多模型并发管理；适合：多模型混合 Serving、边缘/云端统一接口管理
关键性能指标对比（以 LLaMA-3 70B 为参考）：
  • 吞吐量（Throughput）：TensorRT-LLM > SGLang（高 Prefix 命中）≈ vLLM > 原生 HF
  • 首 Token 延迟（TTFT）：SGLang RadixAttention 在 Prefix 命中时极低；TensorRT-LLM 编译优化 TTFT 最低
  • 易用性：vLLM > SGLang > TensorRT-LLM（编译复杂）
生产建议：NVIDIA 硬件 + 追求极致性能 → TensorRT-LLM；开源模型快速上线 → vLLM；多轮对话/RAG 高复用前缀 → SGLang；多模型混合服务 → Triton + 各后端。
💡 核心差异：vLLM = PagedAttention（碎片消除）；SGLang = RadixAttention（前缀复用）；TensorRT-LLM = 编译期算子融合（NVIDIA 极致优化）；Triton = 多后端统一服务层。从 2024 年起，各框架已互相借鉴，边界逐渐模糊，选型以实测性能为准。

CLIP 对比学习原理：InfoNCE 损失 / 图文对齐 / Zero-Shot 分类 / 在多模态大模型中的地位
腾讯 字节 中频
▼

📌 参考答案

CLIP（Contrastive Language-Image Pretraining，OpenAI 2021）核心原理：
  • 双编码器架构：Image Encoder（ViT 或 ResNet）+ Text Encoder（Transformer）分别将图像和文本编码到同一 d 维对齐空间
  • InfoNCE 对比损失：对于 batch 内 N 个图文对，计算 N×N 相似度矩阵；正对（配对图文）相似度最大化，负对（非配对）相似度最小化；损失 = 行方向 CrossEntropy + 列方向 CrossEntropy 的均值
  • 训练数据：400M 从互联网抓取的图文对（WebImageText），无需人工标注
Zero-Shot 分类机制：
  • 将类别名称模板化为文本（"a photo of a {class}"），用 Text Encoder 编码所有类别文本 → 得到文本特征库
  • 将测试图像用 Image Encoder 编码，与文本特征库做余弦相似度，取最高相似度类别
  • 无需任何微调，ImageNet Zero-Shot Top-1 ~76%，与监督 ResNet-50 相当
CLIP 在多模态大模型中的地位：
  • 视觉编码器来源：LLaVA / InternVL / Qwen2.5-VL 等 VLM 大量使用 CLIP ViT（或 SigLIP）作为 Visual Backbone，提供图文对齐的视觉特征
  • Stable Diffusion Text Conditioning：CLIP Text Encoder 编码提示词，作为 UNet Cross-Attention 的 Key/Value，引导去噪方向
  • 检索增强：CLIP 相似度常用于图文跨模态检索（以文搜图、以图搜文）
CLIP 的局限与升级：
  • 细粒度理解弱（分不清"红色的车在蓝色车左边"等关系）→ CLIP 升级到 SigLIP（Sigmoid Loss 替换 Softmax，支持超大 batch），InternViT 进一步用 LLM 对齐
  • 长文本理解弱（Text Encoder 截断 77 token）→ 新一代 VLM 用更长 LLM 做文本侧
💡 一句话：CLIP = 用海量图文对 + InfoNCE 对比损失，将图像和文本编码到同一空间；Zero-Shot 能力来源于文本模板与视觉特征的直接匹配；是几乎所有 VLM 和文生图模型的视觉基础设施。
Q129
SAM / SAM2 视觉基础模型：Promptable Segmentation / Memory Bank / 视频追踪机制
字节 通用 中频
▼

📌 参考答案

SAM（Segment Anything Model，Meta 2023）：
  • 核心范式：Promptable Segmentation——用户输入 Prompt（点击、Box、文本、Mask），模型返回对应分割掩码；一个模型解决任意分割任务，无需针对特定类别训练
  • 三组件架构：① Image Encoder（ViT-H MAE 预训练，重量级，每帧运行一次）提取 Image Embedding；② Prompt Encoder（点/Box/文本映射为 Sparse/Dense Embedding）；③ Mask Decoder（轻量 Transformer，将 Image+Prompt Embedding 解码为掩码）
  • 训练数据：SA-1B，11M 图像 + 10 亿以上掩码（半自动标注 + 人工校正）
SAM2（Meta 2024）：
  • 扩展到视频：在图像 SAM 基础上增加 Memory Bank 机制，将历史帧的分割结果（Object Pointer + Dense Memory Features）存入 Memory Bank
  • Memory Attention：当前帧 Image Embedding 通过 Cross-Attention 查询 Memory Bank，实现目标在视频中的时序追踪（Spatio-temporal Coherence）
  • Memory Bank 结构：Spatial Memory（过去最近 N 帧的帧级特征）+ Object Pointer（每帧目标的轻量 Token 记忆），两者共同参与 Memory Attention
  • 实时性：Image Encoder 用更轻量的 Hiera（分层 ViT），相同精度下速度比 SAM 快 6×；视频追踪可达实时（≥30fps on A100）
在 VLM / 渲染领域的应用：
  • 3D Gaussian Splatting 中用 SAM 做交互式物体分割，实现 3DGS 的语义编辑
  • 视频生成评估：SAM2 追踪生成视频中物体的时序一致性
  • 自动标注管线：代替人工标注大规模分割数据集
💡 SAM vs SAM2 关键差异：SAM = 单图 Promptable Segmentation；SAM2 = SAM + Memory Bank + Memory Attention，将 Prompt 引导的分割扩展到视频时序追踪。Memory Bank 是 SAM2 的核心创新，相当于给分割模型加了"短期记忆"。
